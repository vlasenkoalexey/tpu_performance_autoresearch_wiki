First end-to-end run of the new HuggingFace-`transformers`-based Llama 3 8B
torchax trainer on the **v6e-8 (2 hosts × 4 chips)** GKE slice. Establishes
the torchax-stack performance floor against which the optimization loop
measures. Local v6e-4 single-host smoke is recorded as a sanity check.

## Hypothesis under test

"A torchax + JAX trainer built on HF `LlamaForCausalLM` (meta-init + per-shard
`make_array_from_callback` + `JittableModule` + `torchax.train.make_train_step`
+ `helper.compile_step_func`) can fine-tune Llama 3 8B on wikitext under FSDP
across two TPU hosts via XPK, capture an xprof trace, and produce an
interpretable step-time number." — i.e., **infrastructure check**, not a perf
hypothesis.

## Setup

- Hardware: TPU v6e-8 (1 slice = 2 hosts × 4 chips, topology 2x4),
  GKE cluster `<your-v6e8-cluster>` in `<your-zone>`.
  Per-chip ~31.25 GiB HBM, peak bf16 ~918 TFLOP/s/chip.

- Remat: `jax.checkpoint_policies.nothing_saveable` (recompute everything;

- Command (verbatim, on the pod):


## Baseline comparison

This is the first dated experiment in the program; the only prior reference
on Llama 3 8B is the
[MaxText reference baseline (different stack)](../../maxtext/experiments/2026-04-25-maxtext-llama3-1-8b-v6e8-baseline.md)
at 409.4 TFLOP/s/device, 7,069.7 TPS/device, 44.6 % MFU. That is **not the
target** (MaxText is the reference; torchax is the autoresearch surface) but
indicates the perf ceiling on this hardware shape.

## Results

| Metric | Local v6e-4 single-host | **GKE v6e-8 multi-host (canonical)** | Δ vs local |
|---|---|---|---|
| `batch_size` (per fsdp rank) | 2 | 2 | — |
| `seq_len` | 1024 | 1024 | — |
| Global batch | 8 | **16** | 2× |
| Cold compile (step 0) | 116 s | 92 s | −21 % |
| Warm compile (step 0, cache hit) | 8.8 s | n/a (cache writes denied) | — |
| Steady-state step time | 376 ms | **446 ms** | +18 % |
| Throughput | 21,798 tok/s (5,450/chip) | **36,729 tok/s (4,591/chip)** | +69 % aggregate, **−16 % per-chip** |
| MFU (MaxText formula) | 27.2 % | **22.9 %** | −4.3 pp |
| Loss trajectory (15 steps) | 11.7500 (flat in bf16) | 11.7500 (flat in bf16) | — |

The −16 % per-chip throughput is the multi-host FSDP collective tax (host-to-host
`all-gather` of weights + `reduce-scatter` of gradients). Closing the gap is
the first optimization target after the splash-attention pass.

### MFU formula

Per [maxtext/src/maxtext/utils/maxtext_utils.py `calculate_tflops_training_per_device`](../../../../../raw/code/maxtext/src/maxtext/utils/maxtext_utils.py),
adapted to dense GQA Llama (no MoE / MLA / engram / vision / DPO / MTP):

```
qkv_flops      = 2·B·L·D·(Hq + 2·Hkv)·hd
o_proj_flops   = 2·B·L·D·Hq·hd
ffn_flops      = 6·B·L·Mlp·D                  # SwiGLU: gate + up + down
embed_flops    = 2·B·L·D·V                    # only the lm_head (lookup is gather)
causal_attn    = 2·B·L²·Hq·hd                 # /2 for causal mask
total_per_step = (ffn_flops·nL + (qkv_flops + o_proj_flops)·nL + embed_flops
                  + causal_attn·nL) · 3       # ×3 for 1 forward + 2 backward
```

Llama 3 8B values: `D=4096, Hq=32, Hkv=8, hd=128, Mlp=14336, V=128256, nL=32`.
At v6e-8 batch=2 seq=1024: `total_per_step ≈ 93.8 TFLOPs`, step=446 ms,
peak_per_chip=918 TFLOP/s ⇒ **MFU = 93.8 / 0.446 / 918 = 22.9 %**.

## Observations

1. **Multi-host FSDP collective tax is large at this shape.** v6e-4 single-host
   delivers 5,450 tok/s/chip; v6e-8 multi-host delivers 4,591 tok/s/chip
   (−16 %). At `B=2, L=1024, D=4096`, per-step compute is small relative to
   the inter-host all-gather of `lm_head/embed_tokens` (128k × 4096 bf16 =
   1 GiB, sharded on `fsdp=8`). Closing this gap motivates **larger batch /
   longer seq** and **async-collective scheduling flags** (high-priority).
2. **Compile-cache GCS write permission is denied.** XPK's gcsfuse mount
   (`/data` from `<your-bucket>`) is not writable by the
   pod's UID — JAX prints `PermissionError: [Errno 13] Permission denied`
   per cache entry, so the first call cold-compiles every time (~92 s).
   File system layout / volume mode fix queued as a follow-up — first hit
   would drop time-to-first-step from ~92 s to ~10 s.
3. **bf16 step-precision loss reporting is too coarse.** Loss is reported as
   `11.7500` for every step at `lr=1e-5` — the bf16-stored loss has 8-bit
   mantissa precision (~3 decimal digits), so the per-step `Δloss ≈ 1e-5`
   updates are below the resolution of the printed value. Functional check
   was via local v6e-4 sanity test; full convergence study is **out of scope**
   for the perf loop (model semantics off-limits per program.md).
4. **HF `LlamaForCausalLM` has 2 non-persistent rotary buffers**
   (`model.rotary_emb.inv_freq`, `model.rotary_emb.original_inv_freq`) that
   meta-init leaves unmaterialized; these crash `interop.jax_view` when
   `compile_step_func` traces the train step. Resolved by walking
   `named_buffers(recurse=False)` and replacing each with a replicated
   jax-backed tensor (recompute `inv_freq` from `rope_theta`; zeros for
   unknown leftovers). See `_materialize_buffers_replicated` in `train.py`.
5. **optax `adamw` `count` scalar lands on device-0 only.** Plain
   `optimizer.init(params)` produces a 0-d int32 `count` placed on the
   default device, which collides with `compile_step_func`'s out-shardings
   (`Received incompatible devices for jitted computation`). Resolved by
   walking the opt_state pytree and `jax.device_put`-replicating any leaf
   whose device set is smaller than the mesh.
6. **Compile-time HBM OOM at `batch=4, seq=1024`** on both v6e-4 (4 GiB over,
   35.26/31.25) and v6e-8 (1.05 GiB over, 32.30/31.25). Larger batch is a
   pending hypothesis once memory-saving optimizations land
   (splash attention, scan-over-layers, host-offload of optimizer state).

## Verdict

**Supported** for the infrastructure-check hypothesis: trainer runs end-to-end
on both v6e-4 single-host and v6e-8 multi-host, produces consistent metrics,
captures profile, fits in HBM at `batch=2 seq=1024`. Loss trajectory is
constant-by-bf16-precision but step semantics validated locally; not a
quality concern for the perf loop.

## Next hypotheses generated

Tracked in the program's open-hypothesis list ([README.md](../README.md));
priorities (rough order, expected gain × confidence / effort):

1. **GCS compile-cache write permissions**. Current cold-compile ~92 s wasted
   per run. Fix the gcsfuse `auto_mount`/UID combo (or switch
   `JAX_COMPILATION_CACHE_DIR` to a writable path under `/tmp` + post-run
   sync). Expected gain: time-to-first-step −82 s; effort: S; confidence: high.
2. **Splash attention** (`jax.experimental.pallas.ops.tpu.splash_attention`).
   Replaces the materialized `[B, Hq, L, L]` attention matrix with a
   block-streaming kernel. v6e-8 `bs=2, seq=1024`: attention is ~3 % of
   FLOPs but ~5–10 % of step time per the profile (HBM-bandwidth bound).
   Expected gain: 5–10 % step time and unlocks `seq=2048+`. Effort: M.
3. **`batch=4` on v6e-8** (currently OOMs by 1.05 GiB). After splash lands,
   the activation footprint shrinks enough to fit. Expected gain: better
   per-step amortization of the FSDP collectives → +10–20 % per-chip TPS.
   Effort: S (just bump the flag once memory frees).
4. **Async-collective overlap flags** (`xla_tpu_overlap_compute_collective_tc=true`,
   `xla_tpu_data_parallel_opt_level=L2`). The −16 % per-chip multi-host gap
   suggests under-overlapped FSDP collectives. Expected gain: 5–15 %
   per-chip TPS at this shape. Effort: S.
5. **`seq=2048+` with splash**. Once attention is `O(L)` memory, climb the
   sequence ladder. The program target is `seq=8192`. Effort: S–M.
6. **Scan-over-layers** (`torchax.train.ScannedModule`). The trainer currently
   unrolls 32 layers in HLO (~2.16 M chars per the compile log). Scan
   collapses to one body, reducing compile time and HLO size. Expected gain:
   compile-time only (~5–10× reduction); steady-state throughput likely
   neutral. Effort: M.
7. **`fp32` master weights** (program target eventually). After bf16 perf is
   maxed, switch to `weights_dtype=fp32` with `compute_dtype=bf16` AMP. Will
   require optimizer state to also go fp32 (or remain bf16 if numerically
   sufficient). Effort: M; expected: −15–25 % TPS for accuracy gain (program
   target, not a perf win).
