First MaxText reference run on the autoresearch v6e-8 GKE cluster — wires the MaxText stack (which `program-gke.md` had marked **TODO** on all three GKE clusters) to `<your-v6e8-cluster>` and runs the official `tpu-recipes-v0.1.4` Llama3.1-8B v6e-8 recipe end-to-end. ## Hypothesis under test

The published recipe (`tpu-recipes/training/trillium/Llama3.1-8B-MaxText/v6e-8/llama3-1-8B-1xv6e-8.sh`) reproduces the README-quoted ceiling of **~413 TFLOP/s/device, ~7,139 tokens/s/device** at step 14 on Trillium v6e-8 when run with the locked tag stack (MaxText `tpu-recipes-v0.1.4` + `jax-stable-stack:tpu:jax0.6.1-rev1`) on this cluster. Verifying this turns the recipe into a measured anchor we can compare hand-tuned stacks against, rather than a quoted target.

## Observations

1. **Step time is exceptionally tight** — 3.475 to 3.478 s across 8 steady-state steps (Δ = 3 ms = 0.09 %). Indicates no host-side jitter, no preemption pressure during the run, and a clean steady-state that's straightforward to compare across follow-up experiments.
2. **44.6 % per-chip MFU is materially higher than the AFMv7 3B numbers** captured in [program-gke.md](#) — 27.32 % MFU for AFMv7 3B LoRA DDP-compile (the closest direct comparison on the same cluster). Two factors plausibly drive this: (a) Llama3-8B is a denser compute workload at seq=8192 (more matmul, less collective per FLOP) than AFMv7's 3B at seq=2048; (b) the MaxText recipe ships the splash kernel with `fused_bwd=True` and the host-offload + disable-collective-matmul XLA flag stack tuned for this exact shape. Further attribution requires per-knob ablation (see "Next hypotheses").
3. **HBM headroom is large** — 10.66 GiB peak out of 31.25 GiB available (≈ 34 % used). Suggests `per_device_batch_size=3` is conservative for this offload-heavy config; `bs=4` might be feasible without reducing the offload set, and dropping a subset of the four `*_proj=offload` flags would also fit.
4. **Profiler stop is expensive** — the +5.54 s overhead on step 15 (xplane.pb write to GCS, ~31 MiB) is a single-step blip but visible. Not relevant for this run (we're 4-step-deep in steady-state by then) but a caution for any future run that profiles around step 0 or where step count is small.
5. **Step-0 compile = 12 s** is fast — confirms the AOT-compiled MaxText path with the published flag set caches well; subsequent runs from the same image will benefit from libtpu compile cache. For comparison, gemma4 program runs commonly hit 100+ s on cold compile under torchax.
6. **Fully reproducible against the recipe README within 1 %.** The −1.0 % delta vs the README's 413.4 / 7,138.9 numbers is within typical noise for a single-run comparison; if a follow-up wants to claim parity tighter than that, multi-seed averaging would be needed.

## Verdict + reasoning

**`supported`.**

- Measured throughput (409.42 TFLOP/s/device, 7,069.66 Tokens/s/device steady-state median) reproduces the recipe README's published 413.4 / 7,138.9 numbers within **−1.0 %** — well inside the noise band.
- Loss trajectory is monotone-decreasing (12.264 → 1.792 over 19 steps) and the model trained to similar terminal loss as expected for synthetic data; no divergence, no NaN, no infrastructural failure.
- All 20 steps completed cleanly; both pods finished in `Succeeded` phase. Neither preemption nor RESOURCE_EXHAUSTED nor rendezvous timeout occurred. Profile artifacts (`xplane.pb` + `trace.json.gz`) saved to GCS and mirrored to `raw/profiles/`.
- This satisfies SCHEMA rule 6 ("supported requires measured improvement beyond noise + no semantic change + no regression") for a baseline-establishment context: the comparison target is the recipe README, not a prior run on this cluster.

## Next hypotheses

To be filed as separate `wiki/hypotheses/*.md` pages after the model page is created. In rough rank order (`expected_gain × confidence / effort`):

1. **`maxtext-llama3-8b-v6e8-bs4`** — bump `per_device_batch_size` to 4. Expected: 5–15 % TPS gain (HBM has 66 % headroom; v6e-8 cluster is known to support b=4 stable). Risk: bs ≥ 5 hits `E0200 RuntimeUnexpectedCoreHalt` per program-gke.md gotchas, so b=5 is out, b=4 is the obvious next step. Confidence: high. Effort: S.
2. **`maxtext-llama3-8b-v6e8-drop-projection-offloads`** — ablate each of `out_proj/query_proj/key_proj/value_proj=offload` individually (4 sub-experiments, then a combined "no proj offload" run). Expected: each offload costs some HBM-throughput trade; with HBM headroom we may be able to drop them. Confidence: medium. Effort: M.
3. **`maxtext-llama3-8b-v6e8-toggle-fused-bwd`** — flip `sa_use_fused_bwd_kernel` to False to isolate the fused-backward splash kernel's contribution at v6e-8 scale (the gemma4 program already has a positive read on this kernel at v6e-4 — see [exp 18](../../../gemma4_autoresearch_optimization/jax/experiments/2026-04-23-exp18-fused-bwd-batch3.md)). Confidence: medium-high (consistent across families). Effort: S.
4. **`maxtext-llama3-8b-v6e8-toggle-sparsecore-offload`** — toggle `ENABLE_SPARSECORE_OFFLOADING_FOR_ALL_REDUCE` off to isolate sparsecore contribution at fsdp=8. v6e/Trillium-specific perf knob; clean ablation. Confidence: medium. Effort: S.
5. **`maxtext-llama3-8b-v6e8-toggle-collective-matmul`** — flip `DISABLE_COLLECTIVE_MATMUL` off (the recipe explicitly disables it; that decision is the most fragile in the flag stack). Confidence: medium. Effort: S.
6. **`maxtext-llama3-8b-v6e8-vs-torchax-port`** — once an HF-Llama-3.1-8B torchax port lands at v6e-8, run head-to-head at matching `seq=8192 bs=3 fsdp=8` and measure the cross-stack ceiling gap. Confidence: medium (cross-stack comparisons are noisy). Effort: L (depends on the torchax port being ready). Mirrors the gemma4 program's torchax-vs-JAX comparison structure.
7. **`maxtext-llama3-8b-v6e128-baseline`** + **`-v6e256-baseline`** — port this exact recipe to the larger clusters. Expected per-chip throughput decay with shard count (matches the AFMv7 36.6 → 33.5 → 30.7 % MFU pattern in program-gke.md). Confidence: high (the recipe ships v6e-128 and v6e-256 variants too; these are deterministic ports). Effort: M each.
