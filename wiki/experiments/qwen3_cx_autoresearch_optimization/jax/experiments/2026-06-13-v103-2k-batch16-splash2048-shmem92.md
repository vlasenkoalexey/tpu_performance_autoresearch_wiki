# Qwen3 JAX current v103 - 2k batch16 Splash BKV2048 shmem92

## Hypothesis under test

**Hypothesis**: The BKV2048 branch has a shared-memory threshold: shmem95 tied
the speed frontier but did not improve HBM, while shmem90 created **2.002 GiB**
free HBM but regressed throughput by about **58-63 tok/s**. A midpoint at
**shmem92** may recover some headroom while staying closer to the v095/v096
speed band.

**Mechanism**: Reuse v095/v096 exactly and change only
`--xla_tpu_scheduler_percent_shared_memory_limit=100` to **92**. Keep
`--batch_size=16`, `--seqlen=2048`, scoped VMEM **98304 KiB**, no-scan remat,
activation sharding, Tokamax Splash, MaxText CE, SparseCore reduce-scatter and
all-reduce offload, SparseCore collective aggregator, and:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

**Predicted signal**: Support for speed requires throughput above the confirmed
v095/v096 **70.95k tok/s** band. Support for a robustness branch requires
same-band throughput with materially more HBM headroom. Lower throughput without
useful headroom, compile/runtime failure, or invalid numerics refutes shmem92.

## Verdict

**Refuted for speed; useful threshold/headroom data.** shmem92 gives the same
~2.0 GiB HBM headroom and byte-identical HLO as shmem90, but script throughput
is **70,928-70,929 tok/s**, below the v095/v096 confirmed speed frontier
(**70,959-70,976 tok/s**). The next bracket point is shmem93: it tests whether
the HBM transition starts at 93 while preserving more of the v095/v096 runtime
speed.
