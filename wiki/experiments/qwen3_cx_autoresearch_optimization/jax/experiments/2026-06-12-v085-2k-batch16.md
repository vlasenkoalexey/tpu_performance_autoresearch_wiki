# Qwen3 JAX current v085 - 2k batch16 scale

## Hypothesis under test

**Hypothesis**: v084's per-chip batch **12** improved the 2k frontier to
**69,547 tok/s**, **44.7% script MFU**, and **65.5% MXU**, while retaining
**5.0216 GiB** runtime HBM headroom. Per-chip **batch_size=16**
(`global_batch=128`) should test the near-ceiling point for direct batch
scaling: it may improve MXU further, or it may OOM / lose throughput from memory
pressure and longer non-matmul regions.

**Mechanism**: Reuse v084 exactly, changing only `--batch_size=16`
(`global_batch=128`) at `--seqlen=2048`. Keep no-scan remat, activation
sharding, Tokamax Splash, MaxText CE, scoped VMEM **100352 KiB**,
latency-hiding rerun **3**, SparseCore reduce-scatter/all-reduce offload,
SparseCore collective aggregator, and shared-memory limit **90**.

**Predicted signal**: Support requires clean finite loss plus throughput above
v084's **69,547 tok/s**. OOM, severe profile-step variance, or a near-flat /
lower throughput result refutes further direct batch scaling and should pivot to
scheduler or kernel-shape changes.

## Verdict

Supported and current 2k frontier. Batch scaling from per-chip **12** to **16**
improved best-worker throughput from v084's **69,547 tok/s** to **70,184 tok/s**
(**+0.9%**) and raised MXU from **65.5%** to **66.8%**. The gain is smaller
than prior batch jumps and runtime HBM now has only **1.9936 GiB** free, so raw
batch scaling is near its useful limit. Next probe should keep batch **16** and
adjust compiler memory/scheduling, starting with lower scoped VMEM **98304 KiB**.
