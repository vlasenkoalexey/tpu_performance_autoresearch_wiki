# Qwen3 JAX current v123 - 2k batch16 Splash BKV2048 reduce-scatter latency 4 VMEM98816

## Hypothesis under test

**Hypothesis**: v098 showed `--xla_tpu_scoped_vmem_limit_kib=100352`
selects a distinct but slower HLO, while v122 showed shmem99 is inert. A
close high-side midpoint, `98816`, may alter scheduling enough to improve the
v114/v119 graph without triggering the copy/memory regression seen at 100352.

**Mechanism**: Reuse the v114/v119 graph settings exactly but change:

```bash
--xla_tpu_scoped_vmem_limit_kib=98816
```

Restore `--xla_tpu_scheduler_percent_shared_memory_limit=100` and keep
reduce-scatter latency multiplier 4, all-reduce latency multiplier 2,
SparseCore all-gather offload disabled, aggregator enabled, batch 16, sequence
2048, and Splash BKV/BKV_COMPUTE 2048 fixed.

## Verdict

Refuted. VMEM98816 is graph-active and improves runtime memory headroom, but it
selects a copy-heavy schedule and regresses throughput, step time, and MXU
utilization. This closes the immediate high-side VMEM branch above the
98304 KiB frontier point.
