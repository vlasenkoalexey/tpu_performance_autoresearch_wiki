# Qwen3 JAX current v082 - 2k batch6 scale

## Hypothesis under test

**Hypothesis**: v081's 2k current-stack control reached **65,505 tok/s** with
**56.5% MXU** and **12.296 GiB** runtime HBM free. Increasing per-chip batch
from **4** to **6** should use that headroom to improve device occupancy and
reduce short-context overhead per token while still fitting comfortably.

**Mechanism**: Reuse v081 exactly, changing only `--batch_size=6`
(`global_batch=48`) at `--seqlen=2048`. Keep no-scan remat, activation sharding,
Tokamax Splash, MaxText CE, scoped VMEM **100352 KiB**, latency-hiding rerun
**3**, SparseCore reduce-scatter/all-reduce offload, SparseCore collective
aggregator, and shared-memory limit **90**.

**Predicted signal**: Support requires clean finite loss plus throughput above
the v081 best-worker **65,505 tok/s** and ideally higher MXU than **56.5%**.
Compile OOM, profile step time that cancels the larger token batch, or a larger
custom-call/loop-fusion fraction refutes direct batch scaling.

## Verdict

Supported. Batch scaling from per-chip **4** to **6** improved best-worker
throughput from v081's **65,505 tok/s** to **67,987 tok/s** and raised MXU from
**56.5%** to **61.6%** while retaining almost **10 GiB** runtime HBM headroom.
The next 2k-specific probe should continue the same axis with per-chip
**batch_size=8** before moving to scheduler or kernel changes.
