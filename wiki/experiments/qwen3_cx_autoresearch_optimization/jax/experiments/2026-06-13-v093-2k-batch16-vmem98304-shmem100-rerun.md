# Qwen3 JAX current v093 - 2k batch16 VMEM 98304 shmem100 rerun

## Hypothesis under test

**Hypothesis**: v092 reached **70,606 tok/s**, above v086/v090, but the HLO was
byte-identical to v086. An exact rerun with fresh cache/profile should determine
whether the shmem100 high-water mark is a reproducible runtime effect or normal
run noise.

**Mechanism**: Reuse v092 exactly, changing only workload identity, compile
cache, and profile/HLO output path. Keep `--batch_size=16`, `--seqlen=2048`,
scoped VMEM **98304 KiB**, latency-hiding scheduler rerun **3**, no-scan remat,
activation sharding, Tokamax Splash, MaxText CE, SparseCore reduce-scatter/all-
reduce offload, SparseCore collective aggregator, and
`--xla_tpu_scheduler_percent_shared_memory_limit=100`.

**Predicted signal**: Support requires another clean finite-loss run in the
v092 band, ideally at or above **70,606 tok/s** or with matching XProf step time
near **3743.2 ms**. A result around v086/v087 closes shmem100 as a noise-band
tie because the HLO is unchanged.

## Status

Completed on 2026-06-13 with `EXIT_CODE=0` on both workers.

## Results

Worker0 reported **70,599 tok/s**, **8,825 tok/s/chip**, and **45.4% script
MFU**. Worker1 reported **70,603 tok/s**, **8,825 tok/s/chip**, and **45.4%
script MFU**. Loss stayed finite, ending at **12.0467** on both workers.

XProf run
`2026-06-13-qwen3-jax-v093-2k-bs16-vmem98304-shmem100-rerun/2026_06_13_02_27_06`
shows **3741.4 ms** average step time, **3.8 ms** standard deviation, and
**67.5% MXU**. Top HLO buckets by time were convolution fusion **60,206.935 ms
/ 67.2%**, custom-call **12,133.267 ms / 13.5%**, loop fusion **10,917.732 ms /
12.2%**, data formatting **3,538.837 ms / 3.9%**, and custom fusion
**1,789.932 ms / 2.0%**.

Fresh optimized train-step HLO remained byte-identical to v086/v092:
**19,710,086 bytes / 117,857 lines**, SHA256
`2b5d503bed2876282591e25458d1515c3d63f1a1ad371f4c6c9cb0d9a020218c`, with HLO
memory report total **27.89 GiB** and preallocated temp **22.13 GiB**.

## Verdict

**Supported.** v093 confirms v092's shmem100 result: script throughput remains
above the v086/v087/v088 band, and profile step time improves to **3741.4 ms**
versus v086's **3743.6 ms**. The HLO is unchanged, so this is best described as
a runtime/profile-window improvement rather than a compiled-program change.
Carry shmem100 provisionally as the current 2k batch16 runtime frontier, while
continuing to verify with adjacent one-axis probes.
