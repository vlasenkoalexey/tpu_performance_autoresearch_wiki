# Qwen3 JAX current v086 - 2k batch16 VMEM 98304

## Hypothesis under test

**Hypothesis**: v085 reached **70,184 tok/s**, **45.1% script MFU**, and
**66.8% MXU**, but runtime HBM has only **1.9936 GiB** free and compiled memory
uses **27.89 GiB** with **22.14 GiB** preallocated temp. Lowering
`--xla_tpu_scoped_vmem_limit_kib` from **100352** to **98304** at the same
batch and sequence length may relieve memory pressure or produce a better
near-ceiling schedule.

**Mechanism**: Reuse v085 exactly, changing only
`--xla_tpu_scoped_vmem_limit_kib=98304`. Keep per-chip **batch_size=16**
(`global_batch=128`), `--seqlen=2048`, no-scan remat, activation sharding,
Tokamax Splash, MaxText CE, latency-hiding rerun **3**, SparseCore
reduce-scatter/all-reduce offload, SparseCore collective aggregator, and
shared-memory limit **90**.

**Predicted signal**: Support requires clean finite loss plus throughput above
or at least matching v085's **70,184 tok/s** with lower memory pressure or lower
variance. Lower throughput with no memory benefit refutes the lower-VMEM probe
and keeps v085's **100352 KiB** schedule.

## Verdict

Supported. Lowering scoped VMEM from **100352 KiB** to **98304 KiB** at
2k/context batch16 improved the script summary from v085's **70,184 tok/s** /
**45.1% MFU** to **70,564 tok/s** / **45.4% MFU**, with XProf step time
improving from **3765.2 ms** to **3743.6 ms** and MXU from **66.8%** to
**67.4%**.

The mechanism did **not** reduce runtime HBM: peak/free memory remained
effectively the same as v085 (**29.25 GiB**, about **1.99 GiB free**), and
compiled memory remained **27.89 GiB**. Carry **98304 KiB** as the current
2k batch16 frontier because it is faster, but do not continue blind local VMEM
sweeps for memory headroom. The next hypothesis should target scheduler/codegen
quality at the same batch16 shape, or confirm v086 with a rerun if the next
candidate is high-risk.
