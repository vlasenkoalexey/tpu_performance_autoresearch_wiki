# Qwen3 JAX current v098 - 2k batch16 Splash BKV2048 VMEM100352

## Hypothesis under test

**Hypothesis**: The pre-BKV2048 batch16 VMEM sweep favored scoped VMEM
**98304 KiB** over **100352 KiB**, but BKV2048 changed the train-step HLO and
raised runtime HBM to **31.14 GiB**. Retesting the high-side **100352 KiB**
point on the confirmed BKV2048 stack may select a better schedule or reveal
whether VMEM tuning is exhausted under the new attention tile.

**Mechanism**: Reuse v096 exactly, changing only:

```bash
--xla_tpu_scoped_vmem_limit_kib=100352
```

Keep `--batch_size=16`, `--seqlen=2048`, shmem100, no-scan remat, activation
sharding, Tokamax Splash, MaxText CE, SparseCore reduce-scatter/all-reduce
offload, SparseCore collective aggregator, and the forward Splash tile
overrides:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

**Predicted signal**: Support requires clean finite loss plus throughput above
the confirmed v095/v096 **70.95k tok/s** band, or a same-band run with better
XProf memory/profile behavior. OOM, a return toward the pre-BKV VMEM100352
lower band, or unchanged HLO/profile with no throughput gain refutes carrying
VMEM100352.

## Status

Completed on 2026-06-13 with `EXIT_CODE=0` on both workers.

## Results

Worker0 reported **70,561 tok/s**, **8,820 tok/s/chip**, and **45.4% script
MFU**. Worker1 reported **70,552 tok/s**, **8,819 tok/s/chip**, and **45.4%
script MFU**. Loss stayed finite, ending at **12.0464** on both workers.

The log confirmed the intended Tokamax Splash config:

```text
bq=2048 bkv=2048 bq_dkv=2048 bkv_dkv=2048 fused_bwd=True
```

XProf run
`2026-06-13-qwen3-jax-v098-2k-bs16-vmem100352-splash2048-shmem100/2026_06_13_03_41_20`
shows **3745.7 ms** average step time and **67.3% MXU**, worse than the
v095/v096/v097 **3720-3724 ms** band.

Memory also worsened: XProf reports peak HBM **31.17 GiB / 31.25 GiB**, leaving
only **0.0775 GiB** free, with **2.52%** fragmentation. HLO memory remains
**27.88 GiB**, with preallocated temp **22.13 GiB**.

The optimized train-step HLO changed from the v095/v096/v097 family:
**19,451,153 bytes / 116,052 lines**, SHA256
`ed70db95bbdffbae13742257ec59811d04e4feb7fe8a5f2493f2a02cb020d9de`.
Notable text counts versus v095/v096/v097: `copy` **17231 -> 19182**,
`all-gather` **7072 -> 6999**, and `fusion` **30274 -> 30259**; `custom-call`,
`convolution`, `splash`, and `tokamax` counts are unchanged.

## Verdict

**Refuted.** VMEM100352 is valid but slower and tighter on memory under
BKV2048. It selects a distinct HLO, but the schedule regresses to **70,561
tok/s**, drops MXU to **67.3%**, and reduces runtime free HBM from **0.1018
GiB** to **0.0775 GiB**. Do not carry VMEM100352; keep scoped VMEM **98304
KiB** as the current BKV2048 frontier point.
