# Qwen3 JAX current v094 - 2k batch17 VMEM 98304 shmem100

## Hypothesis under test

**Hypothesis**: v092/v093 confirm the shmem100 batch16 branch at about
**70.60k tok/s**. Batch17 with shmem90 was not durable, but the confirmed
shmem100 runtime shift may make the batch17 shape competitive enough to exceed
the batch16 high-water mark.

**Mechanism**: Reuse v093's supported shmem100 branch, changing only:

```bash
--batch_size=17
```

Keep `--seqlen=2048`, scoped VMEM **98304 KiB**, latency-hiding scheduler rerun
**3**, no-scan remat, activation sharding, Tokamax Splash, MaxText CE,
SparseCore reduce-scatter/all-reduce offload, SparseCore collective aggregator,
and `--xla_tpu_scheduler_percent_shared_memory_limit=100`.

**Predicted signal**: Support requires clean finite loss plus throughput above
v092's **70,606 tok/s** or a profile step-time improvement over v090/v091
batch17. OOM, tight-memory instability, or throughput in the **70.55k** batch17
band refutes retrying batch17.

## Status

Completed on 2026-06-13 with `EXIT_CODE=1` on both workers before step 0.

## Results

The run compiled far enough to emit repeated Tokamax Splash config lines, but
failed while loading `jit_train_step`:

```text
RESOURCE_EXHAUSTED: Error loading program 'jit_train_step': Attempting to
allocate 2.10G. That was not possible. There are 1.02G free.
```

Both workers reported the same failure during the first jitted train-step load.
No valid throughput summary or XProf profile was produced.

## Verdict

**Refuted / invalid.** `batch_size=17` with shmem100 is past the 2k memory edge:
even after compilation, only about **1.02 GiB** HBM remained and the runtime
could not allocate the next **2.10 GiB** buffer. This closes direct batch scaling
above batch16 on the v092/v093 shmem100 branch. Keep v092/v093's batch16
shmem100 frontier and move to 2k-specific codegen/layout probes.
