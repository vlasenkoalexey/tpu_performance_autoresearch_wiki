# Qwen3 JAX current v097 - 2k batch16 Splash BKV2048 shmem95

## Hypothesis under test

**Hypothesis**: v095/v096 confirmed
`SPLASH_BKV=2048 SPLASH_BKV_COMPUTE=2048` as the current 2k frontier, but both
profiles leave only **0.1018 GiB** free HBM. Reducing the scheduler
shared-memory cap from **100** to **95** may choose a more robust schedule while
preserving the BKV2048 throughput gain.

**Mechanism**: Reuse v096 exactly, changing only:

```bash
--xla_tpu_scheduler_percent_shared_memory_limit=95
```

Keep `--batch_size=16`, `--seqlen=2048`, scoped VMEM **98304 KiB**, no-scan
remat, activation sharding, Tokamax Splash, MaxText CE, SparseCore
reduce-scatter/all-reduce offload, SparseCore collective aggregator, and the
forward Splash tile overrides:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

**Predicted signal**: Support requires clean finite loss plus throughput near
the confirmed v095/v096 **70.95k tok/s** band, ideally with more HBM margin or
no XProf profile regression. OOM, a return to the v092/v093 **70.60k tok/s**
band, or a worse profile refutes carrying shmem95.

## Status

Completed on 2026-06-13 with `EXIT_CODE=0` on both workers.

## Results

Worker1 reported **70,965 tok/s**, **8,871 tok/s/chip**, and **45.6% script
MFU**. Worker0 reported **70,958 tok/s**, **8,870 tok/s/chip**, and **45.6%
script MFU**. Loss stayed finite, ending at **12.0467** on both workers.

The log confirmed the intended Tokamax Splash config:

```text
bq=2048 bkv=2048 bq_dkv=2048 bkv_dkv=2048 fused_bwd=True
```

XProf run
`2026-06-13-qwen3-jax-v097-2k-bs16-vmem98304-splash2048-shmem95/2026_06_13_03_26_34`
shows **3722.8 ms** average step time, **67.9% MXU**, and **0.2% idle**. Top
HLO buckets were convolution fusion **60,211.140 ms / 67.5%** and custom-call
**11,696 ms / 13.1%**-class, matching the v095/v096 profile shape.

Memory did not improve: XProf memory profile reports peak HBM **31.14 GiB /
31.25 GiB**, leaving only **0.1018 GiB** free, the same as v095/v096. HLO memory
is also unchanged: total **27.88 GiB**, preallocated temp **22.12 GiB**.

The optimized train-step HLO is byte-identical to v095/v096:
**19,375,297 bytes / 115,424 lines**, SHA256
`a8b0c4b3833fb22bb72597d9e68cdf28fd70e18ba0a695e247b2ebf56ed69bd2`.
Counts match v095/v096: `all-gather=7072`, `all-reduce=165`,
`reduce-scatter=2774`, `async-start=391`, `async-done=391`,
`convolution=3071`, `custom-call=3846`, `copy=17231`, `fusion=30274`,
`dot_general=7301`, `splash=1880`, `tokamax=1`.

## Verdict

**Refuted / tie.** shmem95 is valid and stays in the confirmed BKV2048
throughput band, but it does not beat v095's **70,976 tok/s** high-water mark
and does not improve the HBM risk: runtime free HBM remains **0.1018 GiB**. The
byte-identical train-step HLO confirms that the shared-memory cap did not create
a distinct compiled program here.

Do not carry shmem95. Keep the current frontier at v095/v096:
batch16, sequence length 2048, scoped VMEM **98304 KiB**, shmem100,
`SPLASH_BKV=2048`, and `SPLASH_BKV_COMPUTE=2048`.
