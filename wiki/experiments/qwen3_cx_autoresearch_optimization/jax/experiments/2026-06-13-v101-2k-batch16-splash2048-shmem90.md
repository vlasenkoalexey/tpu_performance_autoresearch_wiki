# Qwen3 JAX current v101 - 2k batch16 Splash BKV2048 shmem90

## Hypothesis under test

**Hypothesis**: v097 showed shmem95 was byte-identical to the v095/v096
BKV2048 frontier and tied throughput, while v100 showed the frontier has almost
no spare HBM for DQ-reduction HLO growth. A stronger shared-memory scheduler cap
at **90** may force a different async collective / Splash staging schedule that
either improves wall time or creates meaningful runtime memory headroom.

**Mechanism**: Reuse the confirmed v095/v096 frontier exactly and change only
`--xla_tpu_scheduler_percent_shared_memory_limit=100` to `90`. Keep
`--batch_size=16`, `--seqlen=2048`, scoped VMEM **98304 KiB**, no-scan remat,
activation sharding, Tokamax Splash, MaxText CE, SparseCore reduce-scatter and
all-reduce offload, SparseCore collective aggregator, and:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

**Predicted signal**: Support requires clean finite loss plus throughput above
the confirmed v095/v096 **70.95k tok/s** band, or same-band throughput with
clear HBM headroom improvement. Lower throughput, unchanged HBM with no profile
gain, compile failure, or invalid numerics refutes carrying shmem90.

## Verdict

Refuted for speed, useful as a headroom branch. v101 is **58-63 tok/s** below
the confirmed v095/v096 speed frontier, so do not replace the best-MFU setting
with shmem90 for batch16. However, the much larger runtime HBM headroom makes it
the right base for one more batch-scaling probe: batch17 with BKV2048.
