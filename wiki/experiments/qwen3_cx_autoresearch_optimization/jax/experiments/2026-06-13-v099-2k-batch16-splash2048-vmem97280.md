# Qwen3 JAX current v099 - 2k batch16 Splash BKV2048 VMEM97280

## Hypothesis under test

**Hypothesis**: v098 showed the high-side scoped VMEM **100352 KiB** schedule is
distinct but slower and more HBM constrained under BKV2048. A low-side
**97280 KiB** retest may improve HBM headroom or select a better schedule below
the current **98304 KiB** frontier point.

**Mechanism**: Reuse v096 exactly, changing only:

```bash
--xla_tpu_scoped_vmem_limit_kib=97280
```

Keep `--batch_size=16`, `--seqlen=2048`, shmem100, no-scan remat, activation
sharding, Tokamax Splash, MaxText CE, SparseCore reduce-scatter/all-reduce
offload, SparseCore collective aggregator, and the forward Splash tile
overrides:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

**Predicted signal**: Support requires clean finite loss plus throughput above
the confirmed v095/v096 **70.95k tok/s** band, or a same-band run with materially
better XProf memory/profile behavior. Lower throughput, OOM, or unchanged
memory/profile refutes carrying VMEM97280 and closes the immediate VMEM bracket
around **98304 KiB**.

## Status

Completed on 2026-06-13 with `EXIT_CODE=0` on both workers.

## Results

Worker1 reported **70,899 tok/s**, **8,862 tok/s/chip**, and **45.6% script
MFU**. Worker0 reported **70,889 tok/s**, **8,861 tok/s/chip**, and **45.6%
script MFU**. Loss stayed finite, ending at **12.0469** on both workers.

The log confirmed the intended Tokamax Splash config:

```text
bq=2048 bkv=2048 bq_dkv=2048 bkv_dkv=2048 fused_bwd=True
```

XProf run
`2026-06-13-qwen3-jax-v099-2k-bs16-vmem97280-splash2048-shmem100/2026_06_13_03_55_34`
shows **3727.5 ms** average step time and **67.7% MXU**, slower than the
v095/v096 frontier.

Memory improved slightly but not decisively: XProf reports peak HBM
**31.13 GiB / 31.25 GiB**, leaving **0.1139 GiB** free, versus **0.1018 GiB**
on v095/v096/v097. HLO memory remains **27.88 GiB**, with preallocated temp
**22.12 GiB**.

The optimized train-step HLO changed from the v095/v096/v097 family:
**19,384,632 bytes / 115,334 lines**, SHA256
`144caf5989b6207f46d0d69929f6adc8b030da0c96b98e066d92098a9dfca227`.
Notable counts: `all-gather=7071`, `all-reduce=165`, `reduce-scatter=2774`,
`async-start=391`, `async-done=391`, `convolution=3071`,
`custom-call=3846`, `copy=17356`, `fusion=30295`, `dot_general=7337`,
`splash=1880`, `tokamax=1`.

## Verdict

**Refuted / memory-only tie.** VMEM97280 is valid and gives a small runtime HBM
margin improvement (**0.1139 GiB** free vs **0.1018 GiB**), but it regresses the
script throughput to **70,899 tok/s** and XProf step time to **3727.5 ms**. That
does not beat or confirm the v095/v096 frontier. Do not carry VMEM97280 for
performance; use it only as a potential emergency robustness fallback.

The immediate VMEM bracket around BKV2048 is closed: **98304 KiB** remains the
current frontier point.
