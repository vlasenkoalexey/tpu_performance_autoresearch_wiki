# Qwen3 JAX current v115 - 2k batch16 Splash BKV2048 reduce-scatter latency multiplier 5

## Hypothesis under test

**Hypothesis**: v113 showed reduce-scatter latency multiplier 2 regressed, and
v114 with multiplier 4 tentatively improved the harness average over the
v095/v096 multiplier 3 frontier without increasing compiled memory. Increasing
the multiplier to 5 may continue to improve reduce-scatter hiding.

**Mechanism**: Reuse v114 exactly but change:

```bash
--xla_tpu_sparse_core_reduce_scatter_latency_multiplier=5
```

Keep all other v114 settings fixed, including batch 16, seq 2048, scoped VMEM
98304 KiB, shmem 100, no-scan remat, activation sharding, Tokamax Splash,
MaxText CE, SparseCore all-reduce and reduce-scatter offload, all-gather ACF,
reduce-scatter ACF, and:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

## Results

- Workload failed before step 0 with `RESOURCE_EXHAUSTED` while loading
  `jit_train_step`.
- Runtime error: attempting to allocate 2.12 GiB with only 1.86 GiB free.
- Optimized train-step HLO:
  `9a8061f2a54e1bb193b8ed78b21a5818cc65184cae7600e21d81c6053b39b350`,
  19,345,298 bytes / 115,222 lines.
- HLO counts: `all-gather=7051`, `all-reduce=165`,
  `reduce-scatter=2774`, `async-start=373`, `async-done=373`,
  `convolution=3078`, `custom-call=3828`, `copy=17383`,
  `fusion=30190`, `dot_general=7281`, `tokamax=1`.
- Compiled memory: 28.25 GiB total bytes used, with 22.50 GiB
  preallocated temp.

## Verdict

Hard refuted. Multiplier 5 pushes the memory envelope past the already tight
runtime load boundary. Do not carry reduce-scatter latency multiplier 5; keep
v114's multiplier 4 as the tentative frontier setting for this axis.
