# Qwen3 JAX current v114 - 2k batch16 Splash BKV2048 reduce-scatter latency multiplier 4

## Hypothesis under test

**Hypothesis**: v113 showed multiplier 2 is too low or no better. Raising
`--xla_tpu_sparse_core_reduce_scatter_latency_multiplier` from 3 to 4 may
prioritize reduce-scatter hiding more aggressively and improve the profile
window on the BKV2048 frontier.

**Mechanism**: Reuse v095/v096 exactly but change:

```bash
--xla_tpu_sparse_core_reduce_scatter_latency_multiplier=4
```

Keep all other v095/v096 settings fixed, including batch 16, seq 2048, scoped
VMEM 98304 KiB, shmem 100, no-scan remat, activation sharding, Tokamax Splash,
MaxText CE, SparseCore all-reduce and reduce-scatter offload, all-gather ACF,
reduce-scatter ACF, and:

```bash
export SPLASH_BKV=2048
export SPLASH_BKV_COMPUTE=2048
```

## Verdict

Tentatively promoted to the 2k frontier. The harness average improves from the
v095/v096 class by roughly 60 tok/s global, and both workers agree. XProf step
time is effectively tied with v095 (3721.2 ms here vs 3720.2 ms there), so the
gain is small and should not be overinterpreted, but the HLO is genuinely
different and the memory envelope remains unchanged. Continue the one-axis
latency-model bracket with reduce-scatter latency multiplier 5.
