# Qwen3 JAX current v092 - 2k batch16 VMEM 98304 shmem100

## Hypothesis under test

**Hypothesis**: v090/v091 showed that batch17 is not a reliable improvement and
that the profile remains dominated by convolution fusion/codegen buckets. At the
stable v086 batch16 shape, raising
`--xla_tpu_scheduler_percent_shared_memory_limit` from **90** to **100** may give
the scheduler more room and improve step time without crossing the 2k HBM limit.

**Mechanism**: Reuse v086 exactly, changing only:

```bash
--xla_tpu_scheduler_percent_shared_memory_limit=100
```

Keep `--batch_size=16`, `--seqlen=2048`, scoped VMEM **98304 KiB**,
latency-hiding scheduler rerun **3**, no-scan remat, activation sharding,
Tokamax Splash, MaxText CE, SparseCore reduce-scatter/all-reduce offload, and
SparseCore collective aggregator.

**Predicted signal**: Support requires clean finite loss plus throughput above
v086's **70,564 tok/s** or XProf step time below **3743.6 ms** without unsafe HBM
headroom. OOM, same HLO with flat/slower throughput, or a tight-memory
regression refutes the shmem100 branch for 2k.

Completed on 2026-06-13 with `EXIT_CODE=0` on both workers.

## Results

Worker0 reported **70,606 tok/s**, **8,826 tok/s/chip**, and **45.4% script
MFU**. Worker1 reported **70,591 tok/s**, **8,824 tok/s/chip**, and **45.4%
script MFU**. Loss stayed finite, ending at **12.0467** on both workers.

XProf run
`2026-06-13-qwen3-jax-v092-2k-bs16-vmem98304-shmem100/2026_06_13_02_13_35`
shows **3743.2 ms** average step time, **3.1 ms** standard deviation, and
**67.4% MXU**. Top HLO buckets by time were convolution fusion **60,181.248 ms
/ 67.1%**, custom-call **12,133.557 ms / 13.5%**, loop fusion **10,918.561 ms /
12.2%**, data formatting **3,538.545 ms / 3.9%**, and custom fusion
**1,842.162 ms / 2.1%**.

Fresh optimized train-step HLO was byte-identical to v086/v087: **19,710,086
bytes / 117,857 lines**, SHA256
`2b5d503bed2876282591e25458d1515c3d63f1a1ad371f4c6c9cb0d9a020218c`, with HLO
memory report total **27.89 GiB** and preallocated temp **22.13 GiB**.

## Verdict

**Supported provisional / needs rerun.** v092 sets a new best observed script
throughput (**70,606 tok/s**) and slightly improves the profile step time over
v086 (**3743.2 ms** vs **3743.6 ms**). However, the optimized HLO is
byte-identical to v086, so the `shmem100` flag did not create an obvious
compiled-program change. Treat the result as a promising high-water mark and
rerun exactly before carrying the setting.
