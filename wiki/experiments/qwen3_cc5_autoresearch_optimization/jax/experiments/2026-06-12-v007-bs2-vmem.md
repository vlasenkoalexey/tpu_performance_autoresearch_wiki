# Qwen3-8B jax v007 — bs2 + splash + scoped-vmem 80 MB

Retry of [v005](2026-06-12-v005-bs2-splash.md) (inconclusive: splash dkv
kernel's 33.84 MB scoped-VMEM need vs 32 MB default limit) with the blueprint
Phase-2 lever `--xla_tpu_scoped_vmem_limit_kib=81920` — the only added
mechanism. Answers the original question: does bs2 fit in HBM with splash?

## Hypothesis under test

**Hypothesis**: With the scoped-VMEM limit raised to 80 MB (v6e physical
96 MB), bs2 + splash either fits in 31.25 GB HBM and lifts tok/s/chip above
the v002 frontier 4,338, or OOMs in HBM by a quantified margin.

**Mechanism**: `LIBTPU_INIT_ARGS="--xla_tpu_scoped_vmem_limit_kib=81920"`
unblocks the 33.84 MB dkv tile; everything else identical to v005
(`--use_splash=True --batch_size=2`, trunk image).

**Predicted signal**: compile passes the VMEM stage. Then either (a) runs:
collective share drops below v002's 29.6%, tok/s/chip > 4,338; or (b) HBM
RESOURCE_EXHAUSTED with vocab-head allocations (`f32[2,2048,151936]` 2.32 GB)
topping the list, ≈ 33 GB total (v004's 43.12 − ~10 GB splash-eliminated
scores).

**Falsification criterion**: HBM OOM (batch axis still blocked → CE is the
remaining unlock), or tok/s/chip ≤ 4,338 + noise → refuted for this rung.

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:latest` (trunk `3ef2b1e`).
- Workload: `alekseyv-qwen3-cc5-jax-v007-bs2vm`.
- Command (diff vs v005: `LIBTPU_INIT_ARGS` added):
  ```bash
  export LIBTPU_INIT_ARGS="--xla_tpu_scoped_vmem_limit_kib=81920"
  python -u train.py --batch_size=2 --seqlen=2048 --train_steps=20 \
      --use_splash=True --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

20 steps clean (exit 0); no VMEM error (the 81920 KiB flag fixed v005's
crash), no HBM OOM. Splash config active on all layers.

| Metric | v002 frontier (bs1) | This run (bs2) | Δ |
|--------|--------------------|----------------|---|
| Step time | 453 ms | 667 ms | +47% (2× tokens) |
| Throughput | 34,705 tok/s (4,338/chip) | **48,758 tok/s (6,095/chip)** | **+40.5%/chip** |
| MFU | 22.3% (23.3% clean-steps) | **31.4%** | **+8–9 pp** |
| Program HBM | 23.03 GiB | **28.13 GiB** | +5.1 GiB (~3.1 GiB headroom left) |

Loss series stable (12.08–12.10, descending) — batch size is a throughput
config, not a model-math change; no semantics check needed beyond stability.

## Verdict

**supported** — +40.5% tok/s/chip, +8–9 pp MFU, no regression (HBM within
budget, loss stable). **Frontier shifts to 667 ms / 48,758 TPS / 31.4% MFU @
(seq 2048, global batch 16).** Flag-only on the trunk image — the recipe
(`--use_splash=True --batch_size=2` +
`LIBTPU_INIT_ARGS=--xla_tpu_scoped_vmem_limit_kib=81920`) becomes the lane's
standard run config (model page updated); no code merge needed.

## Observations

- The scoped-VMEM flag is mandatory lane-wide for bs ≥ 2 with splash
  fused-bwd on v6e (33.84 MB dkv tile vs 32 MB default).
- Collectives (~30%) are now the top optimization target → SparseCore
  collective offload (llama3-jax precedent on v6e: +3.4%) dispatched as v009.
