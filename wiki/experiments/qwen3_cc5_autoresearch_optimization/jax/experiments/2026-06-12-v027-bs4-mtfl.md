# Qwen3-8B jax v027 — bs4 @ seq8192 on the frontier config

Cheap batch-axis probe at the new op-point: [v026](2026-06-12-v026-mt-flags-bs3.md)
dropped HBM peak to 19.66/31.25 GiB (the layout flags shrank the live-set);
bs4 projects ≈ 30.1 GiB — marginal but probe-worthy, launch-only.

## Hypothesis under test

[qwen3-jax-batch-scaling](../../../../hypotheses/qwen3-jax-batch-scaling.md)
at bs4: on the v026 config (scan + offload_attn + splash + chunked CE f32-x
+ 25-flag stack), bs4 @ seq 8192 fits and beats v026's 5,933 tok/s/chip by
> 1% (amortization of fixed per-step costs).

**Diff vs v026**: `--batch_size=4`. Nothing else.

**Predicted signal**: fits (≈ 30.1 GiB projected vs 31.25); tok/s/chip >
5,992. MaxText match bar 6,953 unchanged.

**Falsification criterion**: OOM, or fits but ≤ 5,933 + 1% → refuted
(batch axis closed at bs3 on this substrate; the v025-measured weak
amortization band holds).

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v024-offload` (= latest).
- Workload: `alekseyv-qwen3-cc5-jax-v027-bs4`.
- Single phase: bs4 seq8192, 20 steps, profiled 12–14, HLO dump;
  LIBTPU = the v026 25-flag stack verbatim.

```bash
export JAX_COMPILATION_CACHE_DIR=gs://tpu-pytorch-alekseyv-us-central2/autoresearch/qwen3_cc5/jax_lane_cache
export LIBTPU_INIT_ARGS="--xla_tpu_scoped_vmem_limit_kib=98304 ... [25-flag stack verbatim from v026]"

python -u train.py --model_id=Qwen/Qwen3-8B --use_real_data=False \
  --batch_size=4 --seqlen=8192 --tp_parallelism=1 \
  --train_steps=20 --weights_dtype=bf16 --use_splash=True \

  --profile_start_step=12 --profile_steps=3
```

## Verdict

**supported**

HBM fits with generous margin (23.72 GiB actual vs 30.1 GiB projected; 75.9%
of cap). tok/s/chip = 6,040, clears the PASS bar (5,992) by +0.8%. PASS bar
defined as v026 + 1%; the batch increment delivered +1.8% — narrowly above
threshold. Loss descent clean; no NaN; numerics unchanged.

Diminishing-returns characterization: bs3→bs4 delivers +1.8% vs bs1→bs3's
+4.9% in the old flags regime. The batch axis is near its asymptote on this
substrate — amortization for the dominant buckets (conv fusion, splash,
data-format) is essentially linear with tokens, so per-step gains come only
from the sub-linear RS-fusion and loop-fusion fractions.

**New frontier**: bs4 @ seq8192 = **6,040 tok/s/chip, 34.6% MFU,
5,425.0 ms/step**. Gap to MaxText: −13.1% (down from −14.7% at bs3). The
single largest recoverable bucket is splash (26.7% / 1,448 ms/step), being
tested in [v028 (splash block sweep)](2026-06-12-v028-splash-blocks-bs4.md).
