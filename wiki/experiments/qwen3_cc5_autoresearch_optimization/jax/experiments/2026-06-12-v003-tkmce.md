# Qwen3-8B jax v003 — tokamax streamed cross-entropy

Tests [qwen3-jax-tokamax-ce](../../../../hypotheses/qwen3-jax-tokamax-ce.md)
(`hlo_prefilter: passed`) on `8B/v6e-8`: replace plain softmax CE with
`tokamax.linear_softmax_cross_entropy_loss` (mosaic_tpu), against the
[v000 baseline](2026-06-02-qwen3-jax-v6e8-baseline.md) at seq 2048 bs 1.

## Hypothesis under test

**Hypothesis**: Streaming the CE via the tokamax Pallas kernel (never
materializing `[BL, V=151936]` logits in HBM) reduces peak/transient HBM by
~1.9 GB/chip at bs1/seq2048 (622 MB bf16 logits + 1.24 GB f32 cast copy) and
does not regress step time, unlocking headroom for bs/seq scaling.

**Mechanism**: `--use_tokamax_ce=True` sets `model.skip_lm_head = True`
(forward returns hidden states); loss computes inside the jit via a
`shard_map`-wrapped `tokamax.linear_softmax_cross_entropy_loss(h32, labels,
w32, reduction="sum", implementation="mosaic_tpu")` with
`in_specs=(P('fsdp',None), P('fsdp'), P())`, `psum('fsdp')`, fp32 boundary
cast (bf16 lse accumulation collapses loss — llama3 torchax exp 66 precedent).

**Predicted signal**: the `f32[2048,151936]` allocation (`fusion.6077`) and the
`bf16[2048,151936]` kOutput (`fusion.10289`) disappear from the train_step
buffer assignment; a tokamax CE `custom-call` appears in the op profile; peak
HBM drops ≥1 GB/chip. Step time flat-to-better.

**Falsification criterion**: no HBM reduction vs plain CE, or step-time
regression beyond noise (kernel + fp32-replicated-weight cost exceeds the
materialization saving at this small shape) → refuted.

## Setup

- Hardware: v6e-8 (2 hosts × 4 chips), fsdp=8, tp=1, 1 slice of
  `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v003-tkmce` (fork commit `90cc6c4`).
- Workload: `alekseyv-qwen3-cc5-jax-v003-tkmce`.
- In-workload semantics A/B (deterministic data, seed 0): phase 1 runs 8 steps
  plain CE (reference), phase 2 runs 20 steps `--use_tokamax_ce=True`
  (measured + profiled). First 8 losses must match within bf16/fp32-boundary noise.
- Command:
  ```bash
  python -u train.py --batch_size=1 --seqlen=2048 --train_steps=8 \
      --use_tokamax_ce=False   # phase 1: loss reference
  python -u train.py --batch_size=1 --seqlen=2048 --train_steps=20 \
      --use_tokamax_ce=True --tokamax_ce_impl=mosaic_tpu \
      --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

Both phases exit 0; wall 7.5 min. Kernel confirmed firing (`[ce] tokamax…
impl=mosaic_tpu` + `linear_softmax_cross_entropy_loss_fwd_pallas_mosaic_tpu`
custom-call in HLO). Autotuning consumed steps 0–1 (~112 s each, cache miss on
the `(2048,4096)×(4096,151936)` key); steady state from step 2.

| Metric | Phase 1 (plain CE) | Phase 2 (tokamax mosaic_tpu) | Δ |
|--------|--------------------|------------------------------|---|
| Steady step time | 511.5 ms | **543.7 ms** | **+6.3% (regression)** |
| Throughput | 32,019 tok/s (4,002/chip) | ~30,100 tok/s steady | −6% |
| Program HBM (train_step buffer assignment) | 29.04 GiB (v000 module) | **29.14 GiB** | **+0.1 GiB — no reduction** |

**Loss parity**: consistent +0.005–0.006 offset across all 8 reference steps
(fp32-boundary path difference; no collapse, no divergence — semantics
acceptable, not the exp-66 failure mode).

**Why no HBM win**: the kernel does eliminate the logits materializations
(zero `[2048,151936]` buffers in the v003 module — predicted signal confirmed
structurally), but the **fp32 boundary cast of the replicated lm_head weight
(`f32[4096,151936]` = 2.49 GB/chip) + fp32 hidden cast** add back what the
streamed logits saved (1.86 GB). Net ≈ +0.1 GiB, plus the extra cast/read work
each step explains the +6.3% time.

## Verdict

**refuted** (standalone, at bs1/seq2048 with fp32 boundary casts) — the
falsification criterion "no HBM reduction vs plain CE" fired: 29.14 vs
29.04 GiB, plus a +6.3% step-time regression. NOT `xla-already-fuses` — the
kernel does stream the logits; the regression is self-inflicted by the fp32
input casts (2.49 GB replicated f32 weight). The fork is NOT merged. The
hypothesis is refuted in this form; the bf16-input derivative below inherits
the streaming benefit without the cast cost.

## Observations

- The "fp32 boundary cast REQUIRED" rule from the llama3 lane is
  **chunked_xla-specific** (bf16 lse accumulation collapse). For `mosaic_tpu`
  the kernel accumulates internally; feeding bf16 h/w avoids the 2.49 GB f32
  weight cast entirely. This is the difference between this refutation and
  the llama3 lane's reported tokamax win.
