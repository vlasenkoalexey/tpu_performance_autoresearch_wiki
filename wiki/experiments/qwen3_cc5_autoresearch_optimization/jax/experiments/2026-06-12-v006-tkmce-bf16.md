# Qwen3-8B jax v006 — tokamax CE with bf16 inputs (+ splash)

Tests [qwen3-jax-tokamax-ce-bf16-inputs](../../../../hypotheses/qwen3-jax-tokamax-ce-bf16-inputs.md)
— the derivative of refuted [v003](2026-06-12-v003-tkmce.md) — at the new
splash frontier ([v002](2026-06-12-v002-splash.md), 453 ms / 22.3% MFU).

## Hypothesis under test

**Hypothesis**: Feeding bf16 hidden/weight directly to the mosaic_tpu tokamax
CE kernel (no fp32 boundary cast) keeps the logits streaming (−1.86 GB
materializations) without v003's 2.49 GB f32 replicated weight cast — net
HBM reduction ≥ 1.5 GB and step time ≤ the plain-CE splash frontier (453 ms).

**Mechanism**: Same `shard_map` + `psum('fsdp')` wrapper as v003, minus the
`.astype(jnp.float32)` casts on h and w. The fp32-cast rule was
chunked_xla-specific (bf16 lse collapse); mosaic_tpu accumulates in-kernel.
Both phases run with splash on, so the only diff under test is the CE path.

**Predicted signal**: tokamax fwd custom-call with **bf16 operand layouts**
(v003's had f32); zero `[2048,151936]` buffers AND no `f32[4096,151936]`
allocation; train_step total bytes < 23.03 GiB (the v002 module total).

**Falsification criterion**: loss collapses or offsets beyond bf16 noise vs
plain CE (in-run A/B) → invalid; HBM/step-time not better than plain-CE
splash → refuted.

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v006-tkmce-bf16` (fork from trunk `3ef2b1e`).
- Workload: `alekseyv-qwen3-cc5-jax-v006-tkmce16`.
- In-workload A/B (deterministic data, seed 0), both phases splash-on:
  ```bash
  python -u train.py --batch_size=1 --seqlen=2048 --train_steps=8 \
      --use_splash=True --use_tokamax_ce=False   # phase 1: reference
  python -u train.py --batch_size=1 --seqlen=2048 --train_steps=20 \
      --use_splash=True --use_tokamax_ce=True --tokamax_ce_impl=mosaic_tpu \
      --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

- **Phase 1 (plain CE + splash)**: clean, 8 steps, **453 ms steady /
  ~36,180 tok/s / 23.3% MFU (trainer summary over clean steps 2–7)** —
  independently reproduces the v002 frontier; the cleaner 6-step window shows
  v002's recorded 22.3% was dragged ~1 pp by the profiler-start and GCS-flush
  spike steps in its 18-step average.
- **Phase 2 (tokamax mosaic_tpu, bf16 inputs)**: crash at Pallas jit-trace of
  step 0 — no steps, no profile, no HLO:
  ```
  ValueError: Invalid dtype for `swap`. Ref dtype: float32. Value dtype: bfloat16.
    pallas_mosaic_tpu.py:334, init_logits (backward kernel):
    xw_scratch_ref[...] = x_ref[...] @ w_ref[...]
  ```
  The `[ce] … (bf16 inputs)` routing line printed — dispatch was correct; the
  failure is inside the kernel: its VMEM scratch Ref is hardcoded float32, so
  bf16 operands are rejected at trace time.

## Profile

Not captured — phase 2 crashed at jit-trace before step 0 (allowed exception
per program.md). Phase 1's reference numbers are logged above.

## HLO Dump

Not produced — crash precedes XLA compilation of the phase-2 module.

## Verdict

**invalid** — the configuration cannot run: tokamax `mosaic_tpu`'s backward
kernel does not support bf16 accumulator Refs in the installed version. The
hypothesis's numerics question (is in-kernel accumulation sufficient without
the fp32 boundary cast?) was never reached. Not a refutation of the idea —
a kernel-support gap. Pivot: the same trainer flag surface supports
`--tokamax_ce_impl=chunked_xla`, which is pure XLA (no dtype-restricted
Pallas refs) and was the llama3 lane's accepted bf16-native CE recipe
(exp 66b there). Dispatched as v008 on the same image, no rebuild.

## Observations

- tokamax `mosaic_tpu` CE bwd hardcodes f32 scratch (`xw_scratch_ref`) —
  bf16-input attempts fail at trace time, not silently. Re-check on tokamax
  upgrades.
- Frontier re-measure: splash at bs1/seq2048 is 453 ms / ~36.2k tok/s /
  **23.3% MFU** on clean steps (v002's 22.3% included capture-overhead steps).
