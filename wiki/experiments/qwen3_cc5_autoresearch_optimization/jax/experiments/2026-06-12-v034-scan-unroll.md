# Qwen3-8B jax v034 — scan unroll @ bs4/seq8192

First implementation round of
[qwen3-jax-manual-weight-prefetch](../../../../hypotheses/qwen3-jax-manual-weight-prefetch.md)
(retrospective-2 #1), scoped to the cheapest correct form: **`jax.lax.scan(
..., unroll=N)`** via env `JAX_SCAN_UNROLL`. Unrolling puts N layers in one
loop body, giving the latency-hiding scheduler a within-iteration window to
overlap layer k+1's FSDP weight all-gather under layer k's compute — the
restructuring the libtpu flags (v030–v032) could not produce. A
carry-based explicit prefetch remains the fallback if unrolling is
insufficient (note: carrying gathered weights inflates saved scan carries
under remat — unroll avoids that trap entirely).

## Hypothesis under test

Unroll=2 hides ~half of the 624 ms exposed AG wait (one of each unrolled
pair overlaps); unroll=4 ~three quarters — PASS if either phase > 6,161
tok/s/chip (frontier +2%); the bucket-level check is async-done < 624 ms.

**Diff vs v027**: code = one-line scan unroll knob (fork
`.repo/2026-06-12-v034-wprefetch`); launch = `JAX_SCAN_UNROLL=2` / `=4`.

**Numerics**: CPU smoke — logits **bit-identical at unroll=2**; unroll=4
differs 8.6e-7 (f32 reassociation in the larger fused body — fp noise).
On-device check: unroll=2 loss series must match v027 exactly; unroll=4
within ±0.001.

**Falsification criterion**: both phases ≤ 6,161 AND async-done not
materially below 624 ms → unrolling doesn't unlock the overlap; escalate to
the explicit carry-prefetch design (or close the hypothesis if the profile
shows the scheduler still serializes).

## Results

Attempt 3 (digest-pinned image, gate passed — image confirmed good): **both
phases crashed pre-step-0 with an XLA INTERNAL error**:

```
The async-start expects the shape of operand 0 to match the async shape at
index {0} (bf16[18,2,4,8192,32,128]{...T(8,128)(2,1)} vs
            bf16[18,2,4,8192,32,128]{...T(8,128)(2,1)S(5)})
```

Identical class at unroll=4 (leading dim 9 = 36/4; 18 = 36/2 — the unroll
factor directly shapes the stacked offload buffer). Diagnosis: the
post-optimization pass reassigns the scan-stacked offloaded activation
tensor to host space `S(5)` AFTER the async-collective lowering already
fixed its device-resident shape — a compiler bug in the triple interaction
**scan-unroll ≥ 2 × offload_attn × async-collective-fusion flags**.
unroll=1 (v027) compiles fine; the unroll dimension is what re-triggers the
offload-space assignment ordering.

## Verdict

**inconclusive — blocked by an XLA compiler bug**, hypothesis untested at
unroll ≥ 2 under offload. The blocker is specific: this libtpu build cannot
compose loop unrolling with `pinned_host` offload buffers and async
collective fusion. Escalation path tested next:
[v035](2026-06-12-v035-unroll-noacf.md) drops the async-collective-fusion
flag family (suspected trigger; its standalone value within the v026 bundle
was never isolated) and re-tries unroll=2, with a control arm to price the
dropped flags.
