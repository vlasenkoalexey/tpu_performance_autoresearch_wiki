# Qwen3-8B jax v020 — scan-over-layers at seq 8192

Tests [qwen3-jax-scan-over-layers](../../../../hypotheses/qwen3-jax-scan-over-layers.md)
— the [retrospective](../../../../analyses/2026-06-12-qwen3_cc5-jax-retrospective.md)'s
#1 direction.

## Hypothesis under test

**Hypothesis**: Replacing the 36-iteration layer loop with `lax.scan` over
stacked layer states keeps step time within 2% of the comparable non-scan
config at seq 8192, while cutting compile from ~2×130 s to O(1) and
regularizing memory scheduling (the bs2-@-8192 path).

**Mechanism**: `--use_scan` → `Qwen3ModelScan`: layers built with the
identical RNG stream (bit-identical init at fixed seed — CPU smoke: fwd
parity 8.3e-7, loss parity 1.5e-5 vs the loop model), states stacked once at
init, `jax.lax.scan` over the stacked state with the save_attn-checkpointed
body. Stacked params shard as `(None,) + per-layer-spec` (build_plan
fallback). `JAX_REMAT_SKIP_EVERY` is n/a under scan (uniform body) — the
comparable references are v015 (full save_attn remat, 1475.7 ms) and v019
(skip-every-5, 1389.0 ms).

**Predicted signal**: phase-2 compile (step 0+1 wall) drops from ~260 s to
< 60 s; HLO module count collapses; step time ∈ [1475 ms ± 2%] (matching
v015's uniform-remat config — beating v019's 1389 ms is NOT expected without
skip-every; the scan win is compile + memory shape, judged accordingly).
Watch backward sharding: OOM or layout errors → the known scan trap.

**Falsification criterion**: step time > v015's 1475.7 ms + 2% at the same
uniform-remat config, or sharding breakage that annotations can't fix →
refuted (compile-time win alone insufficient).

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v020-scan`.
- Workload: `alekseyv-qwen3-cc5-jax-v020-scan`.
- Phases (both: seq8192 bs1, splash + chunked CE f32-x + save_attn remat +
  vmem flag; NO SKIP_EVERY in either phase so the A/B isolates scan):
  ```bash
  # phase 1: scan OFF (loop body) — the v015-config reference, 12 steps
  python -u train.py ... --remat_policy=save_attn --use_scan=False --train_steps=12
  # phase 2: scan ON — 20 steps, profiled
  python -u train.py ... --remat_policy=save_attn --use_scan=True --train_steps=20 --profile_start_step=12
  ```

## Results

Both phases exit 0; scan + remat + splash + CE init lines confirmed; stacked
sharding `matched=14 replicated=0`.

| Metric | Phase 1 (loop) | Phase 2 (scan) | Δ |
|--------|-----------------|----------------|---|
| Compile (step 0 / step 1) | 116.8 s / 116.5 s | **34.1 s / 33.5 s** | **−71% (3.4×)** |
| Steady step | 1475.7 ms | **1470.9 ms** | −0.3% |
| Throughput | 44,402 tok/s | 44,554 tok/s | +0.3% |
| MFU | 31.8% | 31.9% | +0.1 pp |
| **Program HBM** | 27.62 GiB (v015 module) | **19.63 GiB** | **−8.0 GiB** |
| Loss parity (steps 0–11) | reference | exact / ±0.0003 | ✓ bit-level at seed |

## Verdict

**supported** — all three pass-bars met (step ≤ ref+2% — actually −0.3%;
compile 3.4× faster; loss parity bit-level). Plus the unplanned 8 GiB HBM
reduction. Merged to trunk as the `--use_scan` opt-in; the perf recipe at
bs1/seq8192 remains v019's SKIP_EVERY=5 (1389 ms beats scan's 1470.9 — scan
trades the skip-rung for memory), but scan becomes the substrate for the
batch-axis push (v021).

## Observations

- Scan's regularized scheduling is worth −8 GiB live-set at seq 8192 — far
  more than the skip-every ladder could buy. Memory-bound shapes should
  default to scan.
- `JAX_REMAT_SKIP_EVERY` is incompatible with scan (uniform body) — the two
  memory/compute knobs trade against each other per shape.
