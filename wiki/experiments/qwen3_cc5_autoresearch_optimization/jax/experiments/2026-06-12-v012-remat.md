# Qwen3-8B jax v012 — selective activation checkpointing

Tests [qwen3-jax-selective-remat](../../../../hypotheses/qwen3-jax-selective-remat.md):
per-layer `jax.checkpoint` with `dots_with_no_batch_dims_saveable`, two-phase —
overhead at the v007 shape (bs2), then the bs3 unlock probe.

## Hypothesis under test

**Hypothesis**: Selective AC cuts the ~6 GiB/bs activation slope enough that
bs3 fits (needs ~6 GiB vs ~3 GiB headroom at v007), at ≤ ~3% step-time
overhead at bs2 — so bs3's amortization gain nets above the 6,095 tok/s/chip
frontier.

**Mechanism**: Env-gated `jax.checkpoint` around each of the 36 decoder
layers (`--remat_policy=selective` →
`jax.checkpoint_policies.dots_with_no_batch_dims_saveable`: matmul outputs
saved, elementwise/norms rematerialized in backward). Numerics-identical
recompute — phase 1 losses must match v007's logged series step-for-step
(same seed, same global batch).

**Predicted signal**: phase 1 (bs2+remat): program HBM well below v007's
28.13 GiB, step time ≤ ~690 ms; phase 2 (bs3+remat): compiles (no OOM) and
tok/s/chip > 6,095.

**Falsification criterion**: bs3 still OOMs with selective AC, or bs2
overhead > 5%, or bs3 tok/s/chip ≤ 6,095 + noise → refuted.

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `<your-cluster>`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v012-remat` (trunk + remat flag; CPU smoke: remat fwd+bwd OK).
- Workload: `alekseyv-qwen3-cc5-jax-v012-remat`.
- Phases (both splash + vmem flag, seed 0):
  ```bash
  # phase 1: overhead + parity at the v007 shape
  python -u train.py --batch_size=2 --seqlen=2048 --train_steps=12 \
      --use_splash=True --remat_policy=selective
  # phase 2: the unlock probe (profiled)
  python -u train.py --batch_size=3 --seqlen=2048 --train_steps=20 \
      --use_splash=True --remat_policy=selective \
      --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

**Phase 1 (bs2 + selective remat)** — ran 12 steps but:

- **NaN from step 4 onward** (steps 2–3 match v007 within 0.0004; then
  loss=nan steps 4–11). v007 at the *identical* shape/seed/data ran 20 steps
  stable — this is NOT a synthetic-data artifact; the remat-as-implemented
  path diverges. Prime suspect: `dots_with_no_batch_dims_saveable`
  interacting with the splash custom_vjp (forward recompute through the
  Pallas call).
- **Overhead +27%**: 847 ms vs v007's 667 ms (38,696 vs 48,758 tok/s; MFU
  24.9% vs 31.4%) — far above the ~2.7% selective-AC table value; consistent
  with the policy rematerializing the splash forward.

**Phase 2 (bs3 + selective remat)** — compile OOM: **31.97 / 31.25 GB
(over by just 741 MB)**. The breakdown is the valuable signal:

| Size | Buffer | What |
|------|--------|------|
| 3.48 GB | `f32[3,2048,151936]` | logits matmul fwd (plain CE — this image has no chunked CE) |
| 3.48 GB | `f32[6144,151936]` | log_softmax fwd |
| 1.74 GB | `bf16[3,2048,151936]` | logits bf16 copy |
| 28 × 144 MB | `bf16[3,2048,12288]` | **checkpointed** layer activations (remat worked as designed here) |

Remat freed the per-layer activations; the un-remat-able vocab head (8.7 GB)
is the bs3 blocker — exactly what chunked CE (~3.5 GB) removes.

## Verdict

**invalid** — phase 1's NaN-vs-stable-baseline is a numerics violation (and
even numerics aside, +27% overhead at bs2 means bs3's amortization gain could
never net-win through this implementation). Phase 2's OOM breakdown is
retained as the decisive observation: with layer activations checkpointed,
**the vocab head is the bs3 blocker → bs3 + chunked CE without remat
(~26.8 GiB projected) is the correct move**, dispatched as v014. The remat
implementation (policy choice / splash interplay) goes back to the bench —
see observation.

## Observations

- [remat-splash NaN + overhead](../../../../observations/qwen3-jax-remat-splash-nan.md):
  per-layer `jax.checkpoint(dots_with_no_batch_dims_saveable)` over layers
  containing the splash custom_vjp produces NaN from step 4 and +27% step
  time. Do not reuse this remat recipe with splash without root-causing.
