# Qwen3-8B jax v062 — bs5 + chunked CE (parity arm)

Semantics isolation for [v061](2026-06-12-v061-2kce8.md): identical config
to the certified [v058](2026-06-12-v058-2kbs5.md)/[v060](2026-06-12-v060-2kbs5val.md)
frontier (bs5, same data stream) with ONLY the CE implementation swapped to
`chunked_xla`. Also measures the pure CE step-overhead at this op-point
(the v003-era "+3.2% at bs2" figure, re-measured on the modern substrate).

## Hypothesis under test

[qwen3-jax-2k-chunked-ce-batch-unlock](../../../../hypotheses/qwen3-jax-2k-chunked-ce-batch-unlock.md)
— semantics leg. **Loss series must match v058/v060's
12.0960 / 12.0969 / 12.0933 within ~1e-3** (same global batch 40, same seed
⇒ same data; the f32-x chunked CE is numerics-preserving per the v008b rule
and the 8k certification lineage). Step time measures CE overhead: v060
1,301 ms + CE-δ.

**Falsification (semantics)**: loss offset > 1e-3 at same-data ⇒ the CE swap
changes numerics at this shape beyond tolerance ⇒ v061's bs8 result is
`invalid` per rule 8, chunked-CE-unlock retired at 2k.

## Results

20 steps clean (exit 0; chunked-CE autotune was cold for the bs5 shape —
steps 0/1 ≈ 91 + 84 s, now cached).

**Numerics (the point of this run): PARITY FAILED.** Same data, same
substrate, only CE swapped — the chunked_xla loss is **systematically
+0.006 higher at every one of 20 steps** (deltas +0.0051…+0.0070, mean
≈ +0.0060; reference v060 series vs this run). The trajectories descend in
parallel, but the offset is stable and reproducible — a real numerical
difference, ~6× over the pre-registered ±1e-3 tolerance.

**Speed**: median steady **1,355.2 ms vs 1,301 ms (v060) = +54 ms (+4.1%)
pure CE overhead** at bs5 (consistent with the ~81 ms / 4% estimated at bs8
from the v061 profile). Throughput 7,554 tok/s/chip (≈38.9% MFU) — chunked
CE at bs5 is strictly worse than plain CE at bs5.

**Interpretation (hypothesis, quantitatively consistent)**: the chunked
kernel pads vocab 151,936 → 153,600 (75 × 2,048 blocks; padded accumulator
`f32[4096,153600]` visible in the v061 HLO). If the 1,664 phantom columns
contribute e⁰ terms to the LSE denominator, predicted offset =
log(1 + 1664/Z) ≈ 0.006–0.009 at random-init logits — matching the
measurement. Implies ~0.9% softmax mass leaks to phantom columns (≈0.99×
scaling of CE gradients). See
[observation](../../../../observations/chunked-ce-padded-vocab-loss-offset.md).

## Verdict

**refuted** (semantics leg) — chunked_xla CE at 2k/bs5 fails same-data loss
parity (+0.006 systematic) AND costs +4.1% step time. Consequences applied:
[v061](2026-06-12-v061-2kce8.md) verdict → **invalid** (rule 8: the +1.5%
is not reportable); hypothesis
[qwen3-jax-2k-chunked-ce-batch-unlock](../../../../hypotheses/qwen3-jax-2k-chunked-ce-batch-unlock.md)
→ **retired** (reopenable only with a padding-masked kernel fix, M effort —
and even then the projected net is ≈ +1.5%). **The certified 2k frontier
remains v058/v060: bs5 + plain CE, 7,871 tok/s/chip / 40.5% MFU.**

⚠️ **Escalation to human**: the same chunked_xla CE (same image, same
kernel) is part of the **certified seq-8192 frontier** (v048/v051 lineage).
This 2k same-data A/B is the lane's first direct plain-vs-chunked parity
measurement; if the padded-vocab interpretation holds, the 8k stack carries
the same ~+0.006 loss offset / ~0.99× CE-gradient scaling vs plain CE. The
8k certification compared chunked-vs-chunked series (internally consistent),
so its *relative* speedups stand, but the *absolute* numerics question is
now open. Human adjudication requested before any 8k re-litigation.
