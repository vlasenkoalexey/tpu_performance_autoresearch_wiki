# Qwen3-8B jax v014 — bs3 + chunked CE f32-x (no remat)

The move [v012's OOM breakdown](2026-06-12-v012-remat.md) points at: bs3's
only blocker is the 8.7 GB plain-CE vocab head; chunked CE
([v011, parity-validated](2026-06-12-v011-cxla-f32x.md)) replaces it with
~3.5 GB. Tests [qwen3-jax-batch-scaling](../../../../hypotheses/qwen3-jax-batch-scaling.md)
at bs3.

## Hypothesis under test

**Hypothesis**: bs3 + splash + chunked CE f32-x fits (projected ≈ 26.8 GiB =
v012-ph2's 31.97 − 8.7 logits + ~3.5 CE) and the bs2→bs3 amortization gain
outweighs the +3.2% CE overhead — net tok/s/chip > the v007 frontier 6,095.

**Mechanism**: Flag-only on the v011 image: `--batch_size=3 --use_splash=True
--use_tokamax_ce=True --tokamax_ce_impl=chunked_xla` + vmem flag. Global
batch 24 (divisible by fsdp=8 ✓).

**Predicted signal**: fits; collective share drops below v007's ~30%;
tok/s/chip > 6,095 ⇒ frontier shift. Risk: CE overhead grows with tokens
(+3.2% at 2048 tok/chip; bs3 = 6144 tok/chip) and could eat the amortization
gain.

**Falsification criterion**: OOM, or tok/s/chip ≤ 6,095 + 1% → refuted (bs
axis closes at bs2 pending a working remat).

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:v6e8-qwen3-8b-jax-20260612-v011-cxla-f32x` (reused).
- Workload: `alekseyv-qwen3-cc5-jax-v014-bs3ce`.
- Command:
  ```bash
  export LIBTPU_INIT_ARGS="--xla_tpu_scoped_vmem_limit_kib=81920"
  python -u train.py --batch_size=3 --seqlen=2048 --train_steps=20 \
      --use_splash=True --use_tokamax_ce=True --tokamax_ce_impl=chunked_xla \
      --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

Compile-time HBM OOM (exit 1, 0 steps): **34.64 / 31.25 GB — over by
3.40 GB.** Both kernels engaged (splash ×28; `[ce] … chunked_xla`). The
projection (26.8 GiB) was wrong: it borrowed v012-ph2's remat-reduced
activation mass; without remat, bs3's full activations + the CE's own
buffers land at 34.64.

Top allocations: 2.34 GB `f32[4096,153600]` CE-bwd dw buffer + 1.17 GB
`bf16[4096,153600]` CE-fwd pad + ~20 × 144 MB `bf16[3,2048,12288]` FFN/FSDP
activation buffers.

## Verdict

**refuted** — bs3 does not fit even with streamed CE; the remaining 3.4 GB
must come from activations (FFN intermediates) or CE restructure
(vocab-sharded dw ≈ −3.2 GB, itself marginal). Combined with v012: the batch
axis at seq 2048 is closed at bs2 pending a working splash-aware remat. Also
note Adam state is already bf16 (v007 buffer table: params/m/v all bf16) —
the `adamw_bf16_state` Phase-1 lever is already in effect, no free memory
there.

## Observations

- Memory ledger at bs3+CE: fixed ≈ 8–9 GB (params+grads+adam, all bf16,
  fsdp=8) + CE 3.5 GB + activations ≈ 22 GB @ 6144 tok/chip. Activation slope
  ≈ 3.6 GB per 1024 tok/chip with splash, no remat.
