# Qwen3-8B jax v001 — per-chip batch 1→4

Tests [qwen3-jax-batch-scaling](../../../../hypotheses/qwen3-jax-batch-scaling.md) on
`8B/v6e-8`: flag-only change (`--batch_size=1` → `4`) on the unmodified baseline
image, against the [v000 baseline](2026-06-02-qwen3-jax-v6e8-baseline.md).

## Hypothesis under test

**Hypothesis**: Raising per-chip batch from 1 to 4 (global 8 → 32) lifts
tok/s/chip above the baseline 3,994 (MFU above 20.5%) with no code change,
because more work per step amortizes fixed per-step collective + dispatch
overhead.

**Mechanism**: Pure trainer flag `--batch_size=4`. No code, kernel, or sharding
change — fsdp=8 mesh unchanged; global batch 32, tokens/step 65,536 (4× the
baseline's 16,384).

**Predicted signal**: collective share (`collective-permute-done`, 21.7% at
v000) drops substantially as the same per-step FSDP collectives amortize over
4× tokens; `convolution fusion` (matmul) share rises from 37.6% toward 50%+;
MXU utilization rises from 19.9%. Same HLO module structure (only batch dim
changes); no new custom_calls.

**Falsification criterion**: tok/s/chip does not rise above the baseline 3,994
beyond noise (±2%), or the run OOMs before any gain → refuted.

## Verdict

**refuted** (for bs4 on the current baseline code). The falsification
criterion explicitly included "OOM before any gain". This is a compile-time
falsification — the run never executed step 0, hence no profile (allowed
exception per program.md "Profile section" rule). Linear scaling of the
activation footprint puts even bs2 at ≈ 54 GB > 31.25 GB, so the batch axis is
**blocked behind activation-transient elimination** (splash + streamed CE),
exactly as the blueprint's "Phase 4 after Phase 3" ordering predicts — the
Phase-4-first override in this experiment's proposal was the wrong call, now
measured. The hypothesis [qwen3-jax-batch-scaling](../../../../hypotheses/qwen3-jax-batch-scaling.md)
stays **open** — re-test after v002/v003 land.

## Observations

- v6e-8 (31 GB/chip) at Qwen3-8B/fsdp=8 has ≈ 15 GB of activation headroom at
  bs1/seq2048; the bs axis multiplies a ~19 GB/bs activation footprint with
  the baseline (no-splash, no-streamed-CE, no-remat) code.
