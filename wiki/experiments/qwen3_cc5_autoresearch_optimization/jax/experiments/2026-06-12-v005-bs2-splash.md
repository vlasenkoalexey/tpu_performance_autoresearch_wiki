# Qwen3-8B jax v005 — bs2 with splash attention

Tests whether splash (landed in [v002](2026-06-12-v002-splash.md), −6 GiB
program HBM at bs1) unlocks bs2 — the batch axis was blocked at bs1 on
pre-splash code ([v004](2026-06-12-v004-bs2.md): 43.12 GB vs 31.25 GB).

## Hypothesis under test

**Hypothesis**: bs2 + splash either fits (and lifts tok/s/chip above the v002
frontier 4,338 by amortizing per-step collectives) or OOMs by < 12 GB,
quantifying exactly how much memory the CE side must recover.

**Mechanism**: Flag-only on the post-v002 trunk image: `--use_splash=True
--batch_size=2`. v004's footprint minus the ~10 GB of bs2 attention-score
buffers splash eliminates ≈ 33 GB — marginal vs 31.25 GB; the probe decides.

**Predicted signal**: if it fits: collective share drops vs v002's 29.6%
(done+start), MXU util rises above 21.9%. If OOM: vocab-head allocations
(`f32[2,2048,151936]` 2.32 GB + grads) top the failure list — pointing at the
CE derivative (v006) as the remaining unlock.

**Falsification criterion**: OOM (cliff still below bs2 with splash), or
tok/s/chip ≤ 4,338 + noise → refuted for this rung.

## Setup

- Hardware: v6e-8, fsdp=8, tp=1, 1 slice of `alekseyv-tpu-v6e8-spot-xpk`.
- Image: `qwen3-8b-jax:latest` (trunk `3ef2b1e`, splash merged).
- Workload: `alekseyv-qwen3-cc5-jax-v005-bs2spl`.
- Command (diff vs v002 phase 2: `--batch_size=2`):
  ```bash
  python -u train.py --batch_size=2 --seqlen=2048 --train_steps=20 \
      --use_splash=True --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```

## Results

Compile-time **scoped VMEM** `RESOURCE_EXHAUSTED` (exit 1, 0 steps, ~3 min):

```
Ran out of memory in memory space vmem
while allocating on stack for %splash_mha_dkv_no_residuals.1
(block_q_dkv=2048, block_kv_dkv=2048)
Scoped allocation with size 33.84M and limit 32.00M exceeded scoped vmem limit by 1.84M.
```

This is **not** the HBM cliff the probe targeted — the run died before memory
assignment of the full program. At bs2 the splash fused-bwd dkv kernel's
scoped-VMEM footprint crosses the **32 MB default scoped limit** (v6e physical
VMEM is 96 MB). Splash config was active and printed (`bq=2048 bkv=1024
bq_dkv=2048 bkv_dkv=2048 fused_bwd=True`).

## Profile

Not captured — compile-time crash before step 0 (allowed exception per

## Verdict

**inconclusive** — the hypothesis's falsification criterion (HBM cliff /
throughput at bs2) was never evaluated; the run hit an orthogonal,
directly-fixable config wall: default `xla_tpu_scoped_vmem_limit_kib` (32 MB)
< 33.84 MB needed by the bs2 dkv tile. The blueprint's Phase-2 known-good
lever (`--xla_tpu_scoped_vmem_limit_kib=81920`, clamped to hw cap; v6e cap
96 MB) is the canonical fix. Retried as v007 with that flag — the only added
mechanism.

## Observations

- On v6e, splash fused-bwd at `bkv_dkv=2048` fits the 32 MB default scoped
  VMEM at bs1 but not bs2 — any bs ≥ 2 splash run on this lane needs the
  scoped-vmem flag (or `SPLASH_BKV_DKV=1024`). Lane-wide operational note.
