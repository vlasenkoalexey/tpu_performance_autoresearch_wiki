# Qwen3-8B jax v006 — remat + splash + bs=4

## Hypothesis under test

**Hypothesis**: Splash removes the `[B,H,S,S]` scores that OOM'd bs=4 (v002, 91.7 G);
with remat+splash, bs=4 fits at seq2048 and beats the 25.1% frontier.

**Mechanism**: `--use_splash=True --use_remat=True --batch_size=4`, image `qwen3-8b-jax:v006-splash`.

**Predicted signal**: no OOM; sane loss; MFU > frontier.

**Falsification criterion**: OOM, NaN loss, or MFU ≤ frontier.

## Setup

- v6e-8 (fsdp=8, tp=1), image `qwen3-8b-jax:v006-splash`, both `[remat]` and
  `[attn] splash kernel ON` confirmed. bs=4/global 32/seq2048. Splash block config
  `bq=2048 bkv=1024 bq_dkv=2048 bkv_dkv=2048 fused_bwd=True`.

## Results

**Crashed at compile/first-step — scoped VMEM OOM in the splash backward kernel.**

| Field | Value |
|-------|-------|
| Crash | `RESOURCE_EXHAUSTED` in **vmem** (not HBM) |
| Op | `splash_mha_dkv_no_residuals` (Pallas backward dkv kernel) |
| Detail | scoped allocation **34.15 M** vs stack limit **32.00 M** — over by just **2.15 M** |
| Cause | backward dkv block sizes too large (`block_q_dkv=2048, block_kv_dkv=2048`) |
| Loss / MFU | none (crashed before step 0) |

**Key positive signal**: splash got **past the v002 HBM OOM** (91.7 G) — it does
relieve HBM pressure (reached backward compilation). The blocker is now a *tunable*
scoped-VMEM-stack overrun by 2.15 M, **not** an HBM-capacity wall and **not** a
kernel-mis-wiring NaN.

## Profile

Absent — crashed before step 0 (profiler never started).

## Verdict

**invalid** — splash-backward scoped-VMEM OOM (2.15 M over the 32 M stack limit),
no steps ran. Splash itself compiled through forward + into backward and cleared
the HBM wall, so the approach is sound; the fix is to **raise the scoped VMEM cap**
(`LIBTPU_INIT_ARGS=--xla_tpu_scoped_vmem_limit_kib=98304`, flag-only, reuses this
image) and/or shrink the splash backward dkv block (`block_kv_dkv=1024`). Re-dispatched
as **v008** with the VMEM-limit flag. Kernel correctness (loss ~12) still to verify
once it fits.
