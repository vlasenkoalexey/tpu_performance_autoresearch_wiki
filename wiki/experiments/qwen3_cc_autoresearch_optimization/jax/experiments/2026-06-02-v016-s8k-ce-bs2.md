# Qwen3-8B jax v016 — seq8192 + splash + CE + bs=2

## Hypothesis under test

**Hypothesis**: At seq8192 the `[B,L,V]` logits are large (bs=1 was the splash-only
ceiling, [v009](2026-06-02-v009-splash-s8k-vmem.md)); CE drops them, letting bs=2
(global 16) fit at seq8192 and beating the seq8192 bs=1 number (30.4% MFU). **This is
where CE should finally pay off** (unlike seq2048, where logits already fit).

**Mechanism**: `--use_remat --use_splash --use_tokamax_ce --tokamax_ce_impl=mosaic_tpu
--batch_size=2 --seqlen=8192` + vmem flag, image `qwen3-8b-jax:v013-ce`.

**Predicted signal**: fits (CE removes the logit wall); loss ~12 stable; MFU > 30.4%
(the seq8192 bs1 number).

**Falsification criterion**: OOM at bs=2 (CE insufficient), NaN loss, or MFU ≤ 30.4%.

## Setup

- **Hardware**: v6e-8 (fsdp=8), `<your-cluster>`, image `qwen3-8b-jax:v013-ce` (absl-guarded CE).
- **Launch** (diff from v009 frontier: `+--use_tokamax_ce --tokamax_ce_impl=mosaic_tpu`, `--batch_size 1→2`):
  ```
  cd /app/trainer && LIBTPU_INIT_ARGS='--xla_tpu_scoped_vmem_limit_kib=98304' \
  JAX_COMPILATION_CACHE_DIR=gs://.../jax_lane_cache ... \
  python -u train.py --model_id=Qwen/Qwen3-8B --use_real_data=False --seqlen=8192 \
    --tp_parallelism=1 --train_steps=20 --weights_dtype=bf16 \
    --use_remat --use_splash --use_tokamax_ce --tokamax_ce_impl=mosaic_tpu --batch_size=2 \
    --profile_dir=<gcs_root> --profile_start_step=12 --profile_steps=3
  ```
- global batch 16, seqlen 8192.

## Results

| Metric | v009 (splash bs1 seq8192) | v016 (splash+CE bs2 seq8192) | Δ |
|--------|---------------------------|------------------------------|---|
| MFU | **30.4%** | 29.5% | **−0.9 pp** |
| tok/s/chip | 5,305 | 5,139 | −3.1% |
| throughput tok/s | 42,440 | 41,111 | −3.1% |
| step time (ms) | ~3,070 | 3,164 | +3% |
| global batch | 8 | 16 | +8 |
| OOM | — | **no** (CE removed the logit wall) | — |
| loss | 12.09→12.07 stable | 12.10→12.06 stable, no NaN | parity |
| exit | 0 | 0 | — |

bs=2 **fits** at seq8192 (splash-only capped at bs=1) — CE confirmed as the memory enabler. But MFU at the doubled batch is **below** the splash-only bs=1 number: the CE streaming/psum overhead plus the extra compute outweigh the occupancy gain. Steady step time ~3,164 ms (one 3,599 ms outlier at step 4). First-call CE autotune ~140 s/step at step0/1 (cached, expected, not a hang).

## Verdict

**Refuted.** Falsification criterion ("MFU ≤ 30.4%") met: 29.5% < 30.4%. CE *does* relieve the seq8192 large-logits memory pressure (bs=2 fits where splash-only was bs=1-bound) — but the predicted throughput payoff did not materialize. Combined with [v014](2026-06-02-v014-splash-ce-bs6.md) (seq2048, also refuted), this **closes the tokamax-CE topic for this lane**: CE is a validated, numerically-correct *memory enabler* that never beats the splash-only frontier on MFU — useful only if a future config is hard memory-bound on logits and has no cheaper relief. Loss healthy, semantics intact.
