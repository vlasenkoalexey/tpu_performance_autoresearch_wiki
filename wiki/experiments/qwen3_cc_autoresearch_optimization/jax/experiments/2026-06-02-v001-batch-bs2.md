# Qwen3-8B jax v001 — batch bs=2

## Hypothesis under test

**Hypothesis**: Per-chip batch 2 (global 16) lifts MXU occupancy above the
baseline ~20% with no code change.

**Mechanism**: Flag-only `--batch_size=2`; global_batch = 2 × fsdp(8) = 16 at seq2048.

**Predicted signal**: MXU + tok/s/chip rise vs baseline 3,994.

**Falsification criterion**: no improvement beyond ±2%, or OOM.

## Setup

- v6e-8 (fsdp=8, tp=1), BASE_IMAGE `qwen3-8b-jax:latest`, flag-only `--batch_size=2`.
- global_batch 16, per-chip 2, seq2048, bf16, 20 steps.

## Results

**OOM at compile time — never reached step 0.** Container exit 1, ~2.5 min wall.

| Metric | Value |
|--------|-------|
| HBM required | **43.12 G** / 31.25 G per chip — over by **11.87 G** |
| HLO-temp share | **100%** (43.11 G) — fragmentation only 0.7% (genuine capacity overflow) |
| Dominant alloc | `bf16[2,32,2048,2048]` (512 M each) — many remat copies of the full QK^T scores |
| Steps / loss / MFU | n/a (never executed) |

## Profile

Absent — compile OOM before step 0; profiler never started.

## Verdict

**invalid** — compile-time OOM, no run. Combined with [v002 (bs=4)](2026-06-02-v002-batch-bs4.md),
this establishes that **batch-scaling is refuted for the current jax path**: even
bs=2 OOMs at seq2048. Two compounding causes: (1) the dense XLA-SDPA path
materializes the `[B,H,S,S]` scores, and (2) the minimal jax trainer has **no
gradient rematerialization** (the 43 G is 100% HLO-temp). The frontier stays at
the [baseline](2026-06-02-qwen3-jax-v6e8-baseline.md) bs=1 / 20.5% MFU. **Memory
wins must come before batch**: splash attention + gradient remat.
