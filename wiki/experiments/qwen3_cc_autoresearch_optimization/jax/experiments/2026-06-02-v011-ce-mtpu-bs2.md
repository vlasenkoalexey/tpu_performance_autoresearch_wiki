# Qwen3-8B jax v011 — tokamax CE mosaic_tpu (canary, bs=2)

## Hypothesis under test

**Hypothesis**: With `mosaic_tpu`, the streamed CE is numerically correct and ~neutral
at bs=2 — the canary before stacking with splash.

**Mechanism**: `--use_tokamax_ce --tokamax_ce_impl=mosaic_tpu`, remat, bs=2, image `qwen3-8b-jax:v010-ce`.

## Results

**Crashed at compile — absl flags collision (not a CE numerics issue).**

| Field | Value |
|-------|-------|
| Error | `absl.flags._exceptions.UnrecognizedFlagError: Unknown command line flag 'model_id'` |
| Where | tokamax `config.py:63` `flags.FLAGS(sys.argv)` (lazy config resolution inside the mosaic_tpu CE op) |
| Cause | tokamax re-parses the process argv with its absl parser; the trainer uses `fire.Fire`, so argv carries `--model_id=…` which absl rejects |
| Steps / loss / MFU | none (trace-time crash, before step 0) |

## Profile / HLO
Absent — crashed before any XLA compile/dump.

## Verdict

**invalid** — argv-parsing collision, NOT a tokamax shape/numerics error. The CE
kernel never ran (canary inconclusive). **Root fix (code)**: pre-parse absl FLAGS
with only `argv[0]` at trainer startup so tokamax's lazy `flags.FLAGS(sys.argv)`
short-circuits — the exact guard the llama3 jax/torchax trainers carry, which the
qwen3 trainer was missing. Added to `train.py`; rebuilt as image `v013-ce`.
Re-dispatched as **v013** (CE canary) — and v012 (stacked, also CE) will crash the
same way and be superseded by **v014** on the fixed image.
