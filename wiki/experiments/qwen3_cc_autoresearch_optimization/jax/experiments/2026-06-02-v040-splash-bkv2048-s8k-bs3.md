# Qwen3-8B jax v040 — splash bkv=2048 @ seq8192 bs3

## Hypothesis under test

**Hypothesis**: MaxText's qwen3-8b reference uses `sa_block_kv = 2048` (all splash blocks 2048); our frontier
uses `bkv=1024` (bq already 2048). Matching MaxText's all-2048 splash config at the maxtext-CE frontier (v035,
no offload) lifts throughput above 6,030 (larger KV blocks → fewer kernel-boundary crossings / better MXU on
the attention kernel).

**Mechanism**: image `qwen3-8b-jax:v033-maxtext-ce`, v035 config (no offload) + env `SPLASH_BKV=2048
SPLASH_BKV_COMPUTE=2048` (the only two not already 2048). NO code change. Base 12-flag bundle (isolate the
splash-block variable vs v035).

**Predicted signal**: tok/s/chip > v035 6,030; loss parity; exit 0.

**Falsification criterion**: ≤ 6,030 beyond noise (bkv=2048 doesn't help at seq8192 — consistent with the
pre-CE block sweeps v021/v022 being neutral; our bkv=1024 is already near-optimal). VMEM-OOM at the larger
block (then bkv=1024 is the vmem-feasible choice).

## Setup

- Image `qwen3-8b-jax:v033-maxtext-ce`, v035 config (no offload) + env `SPLASH_BKV=2048 SPLASH_BKV_COMPUTE=2048`
  + base 12-flag bundle. `[splash] bq=2048 bkv=2048 bq_dkv=2048 bkv_dkv=2048 fused_bwd=True` confirmed (bkv now
  2048, matching MaxText); `[scan]/[remat:nothing_saveable]/[ce]` ON; `per_chip_batch=3 global_batch=24`.

## Verdict

**Refuted** (parity, not a win). Matching MaxText's `sa_block_kv=2048` is neutral at seq8192/bs3 — our
`bkv=1024` (the v009 autotune winner) was already optimal; the splash kernel is not the bottleneck (it's
already ~92 ms faster than MaxText's per the v035 profile). **seq8192 frontier stays v035 (34.6% / 6,030).**
This was the **last cheap config-level delta vs MaxText** — combined with v038 (flags parity), v039 (offload
refuted), and the sharding match, the **config/flag search space vs MaxText is now exhausted**. The residual
~13% to MaxText is its host-offload pipelining (a runtime/build wall our stack can't reproduce, v039) — out
of config scope.
