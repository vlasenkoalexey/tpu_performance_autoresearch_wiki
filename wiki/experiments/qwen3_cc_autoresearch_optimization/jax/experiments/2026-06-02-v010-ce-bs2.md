# Qwen3-8B jax v010 — tokamax CE (canary, bs=2)

## Hypothesis under test

**Hypothesis**: The streamed tokamax CE is numerically correct and ~neutral on MFU
at bs=2 — the canary before stacking CE with splash for bigger batch.

**Mechanism**: `--use_tokamax_ce=True --tokamax_ce_impl=chunked_xla`, remat, bs=2, seq2048, image `qwen3-8b-jax:v010-ce`.

**Predicted signal**: loss ~12 stable; MFU ≈ 22.0%.

**Falsification criterion**: NaN/garbage loss, crash, or large MFU regression.

## Setup

- v6e-8 (fsdp=8, tp=1), image `qwen3-8b-jax:v010-ce`, `--use_tokamax_ce=True --tokamax_ce_impl=chunked_xla --use_remat --batch_size=2`.

## Results

**Crashed at compile — tokamax API rejected the implementation name.**

| Field | Value |
|-------|-------|
| Error | `ValueError: Unsupported implementation: chunked_xla` (tokamax `linear_softmax_cross_entropy_loss/api.py`) |
| Cause | the image's tokamax accepts only `Literal["mosaic_tpu", "xla"]`; `chunked_xla` was the llama3 *local-source* impl, absent from this pip/image build |
| Steps / loss / MFU | none (compile-time crash, before step 0) |

**Not a CE wiring or numerics bug** — the shard_map/fp32-cast path is fine; only the
impl string was wrong.

## Profile
Absent — crashed before step 0. No HLO (pre-dump).

## Verdict

**invalid** — wrong tokamax impl name (`chunked_xla` not in this build). Fix is
flag-only: `--tokamax_ce_impl=mosaic_tpu` (the streamed Pallas impl that saves the
`[B,L,V]` logits). **Re-dispatched as v011** (mosaic_tpu, same image). The trainer
default should be changed chunked_xla → mosaic_tpu in a later code commit.
