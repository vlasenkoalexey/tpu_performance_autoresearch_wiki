# Qwen3-8B jax v043 — tp=2 activation sharding @ seq8192 bs3

## Hypothesis under test

**Hypothesis**: `--tp_parallelism=2` shards the MLP/attention activations on the `tp` axis, which (per the v035
profile) could shrink the per-chip SiLU recompute inflating our loop-fusion bucket → lift the frontier above
v035 (34.6% / 6,030). FLAG-ONLY (sharding plan already encodes Megatron TP; CPU tp=2 smoke validated tracing).

**Mechanism**: image `qwen3-8b-jax:v033-maxtext-ce` (= v035 frontier) + `--tp_parallelism=2`. Mesh becomes
`fsdp=4 × tp=2`. All four levers (scan/splash/remat-nothing_saveable/maxtext-CE) ON.

**Predicted signal**: tok/s/chip > 6,030 if tp's activation sharding cuts the recompute.

**Falsification criterion**: ≤ 6,030 beyond noise.

## Setup

- Image `qwen3-8b-jax:v033-maxtext-ce`, `--use_remat --use_splash --use_scan --use_maxtext_ce --batch_size=3
  --seqlen=8192 --tp_parallelism=2`, LIBTPU bundle `--xla_tpu_scoped_vmem_limit_kib=98304` + experimental
  scheduler + async-collective-fusion set. Startup confirmed: `[mesh] fsdp=4 tp=2`, `[scan]/[attn]/
  [remat:nothing_saveable]/[ce]` ON, `matched=14 replicated=0`, **`per_chip_batch=3 global_batch=12`**.

## Results — REGRESSION, but CONFOUNDED (global batch halved)

| run | mesh | global_batch | tok/s/chip | MFU | steady step (ms) |
|-----|------|--------------|-----------|-----|------------------|
| v035 | fsdp=8 tp=1 | **24** | 6,030 | 34.6% | 4,075 |
| **v043** | **fsdp=4 tp=2** | **12** | **4,303** | **24.7%** | **2,856** |

Loss 12.0990 → 12.0670 monotone, no NaN, exit 0, 20 steps clean. Compile 46.9 s (cold) → 3.0 s.

**The regression is the batch confound, not tp.** `batch_size` is per-FSDP-shard; tp=2 drops fsdp 8→4, so
global batch = 3×4 = **12** (half of v035's 3×8 = 24). Step time fell to 0.70× (2,856 vs 4,075) but useful
tokens fell to 0.50×, so MFU = 0.5/0.70 × 34.6% ≈ 24.7% ✓. **This run does not isolate the tp lever** — it
conflates tp with a 2× batch reduction.

**Deeper finding (invalidates the tp thesis a priori):** at *equal* global batch, tp cannot reduce per-chip
activation — it trades batch-sharding for model-sharding. At `fsdp=4×tp=2, bs=6` the per-chip MLP activation
`bf16[6,8192,6144]` equals v035's `bf16[3,8192,12288]` (same bytes → same SiLU recompute). tp only reshapes
the matmul N tile (12288→6144) and adds all-reduce collectives after o/down_proj. So tp is unlikely to shrink
the recompute gap. The clean control is **v043b = tp=2 at bs=6** (global batch 24), dispatched 2026-06-05.

## Verdict

**Inconclusive** — the run executed cleanly and proves tp=2 *works* (traces, GQA-splash over tp-sharded heads,
maxtext-CE over tp-sharded vocab, loss stable), but the headline regression is a **batch-halving confound**, not
a tp effect, so it cannot support or refute the hypothesis. Superseded by **v043b (tp=2 bs=6, equal global
batch)** as the clean control. The a-priori activation-invariance argument predicts v043b will be
neutral-to-negative.
