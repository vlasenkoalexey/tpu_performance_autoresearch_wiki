# Qwen3-8B jax v007 — remat + splash @ seq8192

## Hypothesis under test

**Hypothesis**: Splash + remat reach the program-target seq 8192 (bs=1) — the seq
the dense path can't (its `[B,H,8192,8192]` scores OOM) — at MFU > the seq2048 frontier.

**Mechanism**: `--use_splash --use_remat --batch_size=1 --seqlen=8192`, image `qwen3-8b-jax:v006-splash`.

**Predicted signal**: no OOM at seq8192; loss ~12; MFU > 25.1%.

**Falsification criterion**: OOM, NaN, or MFU ≤ 25.1%.

## Setup

- v6e-8 (fsdp=8, tp=1), splash+remat ON, seq8192/bs1/global8. **No `scoped_vmem_limit`
  flag set** (ran on the default 32 M limit — the mistake fixed in v009).

**Crashed — splash-backward scoped-VMEM OOM (default 32 M limit).**

| Steps / loss / MFU | none (crashed before step 0) |
| Infra | minor spot churn on worker-0 (~2m23s); crash was VMEM, not preemption |

## Verdict

**invalid** — same splash-backward VMEM overrun as [v006](2026-06-02-v006-splash-bs4.md),
here only 988 K over because I forgot to set `--xla_tpu_scoped_vmem_limit_kib` on this
run (default 32 M). The seq8192 target number was not obtained. **Re-dispatched as v009**
with `LIBTPU_INIT_ARGS=--xla_tpu_scoped_vmem_limit_kib=98304` (same fix as v008). If the
raised cap still doesn't fit, shrink the splash backward dkv blocks (`block_kv_dkv=1024`)
or disable `fused_bwd`.
