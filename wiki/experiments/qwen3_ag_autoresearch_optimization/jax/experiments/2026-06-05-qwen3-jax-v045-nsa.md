# v045: Native Sparse Attention (NSA) at Batch Size 24

## Hypothesis under test
- **Mechanism**: Use `ejkernel`'s `native_sparse_attention` XLA backend instead of Splash to reduce memory usage during compilation and allow batch size 24 at sequence length 2048.
- **Predicted signal**: Compilation succeeds and MFU improves due to increased batch size.

## Results
- **Outcome**: `RESOURCE_EXHAUSTED` (XLA:TPU compile permanent error).
- **Details**: `Used 396.07G of 31.25G hbm. Exceeded hbm capacity by 364.82G.`

## Analysis
The hypothesis that NSA via XLA fallback reduces HBM usage is FALSE.
- The original XLA fallback implementation used `jax.vmap` over the sequence length, which XLA expanded into a massive graph requiring ~396GB during compilation.
- Attempts to rewrite the forward pass using dense masked chunking via `jax.lax.scan` reduced forward-pass compilation memory, but JAX auto-differentiation still materialized and saved the full chunked attention scores (6GB per layer) for the backward pass, leading to the exact same ~396GB OOM.
- **Conclusion**: NSA on TPU strictly requires a custom Pallas kernel (similar to FlashAttention) that recomputes scores in the backward pass to avoid materializing the full attention matrix. Without such a kernel, NSA cannot run at `bs=24`.

## Verdict
**REFUTED**. XLA fallback for NSA is not memory-efficient enough for TPU v6e at these scale parameters. `bs=16` remains the practical ceiling for the JAX lane with current kernels.
