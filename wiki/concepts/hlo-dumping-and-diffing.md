HLO dumping captures the XLA compiler's intermediate representation at various optimization stages, and HLO diffing compares those stages to reveal what the compiler did (or failed to do). This is the primary diagnostic workflow for fusion analysis, kernel-replacement hypothesis pre-filtering, sharding verification, and offline HLO analysis without TPU access.

## Definition

An **HLO dump** is a text (`.hlo`) or proto (`.pb`) serialization of the XLA High-Level Optimizer IR at a specific compiler pass. XLA's `--xla_dump_to` flag writes one file per pass per module. **HLO diffing** is the comparison of two stages (e.g., `before_optimizations` vs `after_optimizations`) to see what transformations XLA applied — fusions formed, collectives rewritten, layouts assigned, dead code eliminated.

## Why it matters for TPU perf

- **Fusion verification**: confirm whether XLA fuses your target ops into a single kernel (`kind=kOutput` Mosaic) or leaves them as separate HBM round-trips. This is the basis of the [AOT pre-filter](aot-compilation.md) for kernel-replacement hypotheses.
- **Sharding diagnosis**: verify that `sharding=` annotations in the HLO match your intended `PartitionSpec`. Mismatches cause silent all-gathers.
- **Collective placement**: check whether async collectives (`async-start`/`async-done`) are placed where the [latency-hiding scheduler](latency-hiding-scheduler.md) can overlap them with compute.
- **Offline analysis**: when a customer sends dump files, the xprof-mcp dump tools analyze them on CPU — no TPU or xprof server required.
- **Regression diagnosis**: diff the HLO before and after a code change to find which fusion broke or which new collective appeared.

## How to capture dumps

### JAX

Set `XLA_FLAGS` before running:

```bash
# Full dump — all passes, text format
XLA_FLAGS="--xla_dump_to=/tmp/hlo_dump --xla_dump_hlo_as_text" \
  python -m train --steps 1

# Selective dump — only specific passes (faster, smaller)
XLA_FLAGS="--xla_dump_to=/tmp/hlo_dump --xla_dump_hlo_as_text \
  --xla_dump_hlo_pass_re='.*fusion.*|.*layout.*'" \
  python -m train --steps 1

# Final-only dump (smallest — just the optimized HLO)
XLA_FLAGS="--xla_dump_to=/tmp/hlo_dump --xla_dump_hlo_as_text \
  --xla_dump_hlo_pass_re='after_optimizations'" \
  python -m train --steps 1
```

Alternatively, use the [AOT compilation](aot-compilation.md) pipeline to capture HLO without running a full training step:

```python
lowered = jax.jit(fn).lower(*args)
hlo_text = lowered.as_text()                          # pre-optimization
compiled = lowered.compile()
# Post-optimization HLO requires dump files or xprof
```

### PyTorch/XLA

```bash
XLA_FLAGS="--xla_dump_to=/tmp/hlo_dump --xla_dump_hlo_as_text" \
  python train.py
```

PyTorch/XLA dumps use the same `XLA_FLAGS` surface. The HLO module names differ (typically `SyncTensorsGraph.*` vs JAX's `jit_<fn_name>.*`).

## File formats and naming

| Extension | Format | Use |
|---|---|---|
| `.hlo` | Human-readable text | Grep, visual inspection, diff |
| `.pb` | Protocol buffer | Programmatic analysis, xprof ingestion |
| `.html` | Rendered HTML graph | Browser-based visualization |

**File naming convention** (auto-generated by XLA):

```
module_<NNNN>.<jit_fn_name>.cl_<hash>.<stage_name>.hlo
```

Example:
```
module_0262.jit_train_step.cl_854318611.after_optimizations.hlo
```

For a typical training step, XLA produces one module per `jit`-traced function. With scan-over-layers, a single `jit_train_step` module covers the entire step; without scan, you may see one module per layer.

## Stage aliases

Key stages in XLA's optimization pipeline:

| Stage name | What it shows |
|---|---|
| `before_optimizations` | Raw HLO from the frontend (JAX/PyTorch/TF) before any XLA pass |
| `after_optimizations` | Fully optimized HLO — fusions formed, layouts assigned, collectives placed |
| `spmd_partitioner` | HLO after GSPMD sharding propagation (reveals actual per-device ops + collectives) |
| `fusion` / `instruction_fusion` | HLO after the fusion pass specifically |
| `layout_assignment` | HLO after layout decisions (memory order of tensors) |
| `latency_hiding_scheduler` | HLO after the scheduler reorders ops for collective overlap |

Use `--xla_dump_hlo_pass_re` to capture only the stages you need. The most common comparison is `before_optimizations` vs `after_optimizations`.

## xprof-mcp dump tools

These tools work entirely on CPU — no TPU or xprof server required. Point them at a directory containing `.hlo` or `.pb` dump files.

| Tool | Purpose | Key parameters |
|---|---|---|
| `list_hlo_dump_modules` | List all modules in a dump directory | `dump_dir` |
| `get_hlo_dump` | Read a specific module at a specific stage | `dump_dir`, `module_id`, `stage` |
| `diff_hlo_stages` | Diff two stages of the same module (or two modules) | `dump_dir`, `module_id`, `stage_a`, `stage_b` |
| `get_hlo_dump_neighborhood` | Extract the neighborhood of a specific HLO op (its producers/consumers) | `dump_dir`, `module_id`, `op_name` |

### Usage example

```python
# 1. List available modules
list_hlo_dump_modules(dump_dir="/tmp/hlo_dump")
# → module_0262.jit_train_step.cl_854318611

# 2. Read post-optimization HLO
get_hlo_dump(
    dump_dir="/tmp/hlo_dump",
    module_id="module_0262",
    stage="after_optimizations"
)

# 3. Diff before vs after to see what XLA did
diff_hlo_stages(
    dump_dir="/tmp/hlo_dump",
    module_id="module_0262",
    stage_a="before_optimizations",
    stage_b="after_optimizations"
)

# 4. Inspect neighbors of a specific op
get_hlo_dump_neighborhood(
    dump_dir="/tmp/hlo_dump",
    module_id="module_0262",
    op_name="fusion.316"
)
```

## Key HLO patterns to search for

When inspecting dumps, grep for these patterns:

| Pattern | What it means | Command |
|---|---|---|
| `kind=kOutput` | Mosaic/Pallas kernel — XLA's fused TPU kernel | `grep 'kind=kOutput' *.hlo` |
| `custom-call` | External kernel (Pallas, Mosaic, or platform-specific) | `grep 'custom-call' *.hlo` |
| `all-reduce` / `all-gather` / `reduce-scatter` | FSDP/TP collective operations | `grep -E 'all-reduce\|all-gather\|reduce-scatter' *.hlo` |
| `async-start` / `async-done` | Async collective (good — enables overlap) | `grep 'async-start' *.hlo` |
| `sharding=` | Per-tensor sharding annotation | `grep 'sharding=' *.hlo` |
| `convolution` | Matmul (XLA represents matmuls as convolutions on TPU) | `grep 'convolution' *.hlo` |
| `dynamic-slice` / `dynamic-update-slice` | KV-cache or scatter/gather ops | `grep 'dynamic-slice' *.hlo` |
| `fused_computation` | Named fusion body — read to see which ops XLA merged | `grep 'fused_computation' *.hlo` |
| `bytes_accessed` | Cost-model bytes for an op | `grep 'bytes_accessed' *.hlo` |

### Fusion verification recipe

To check whether XLA already fuses ops A and B (e.g., RMSNorm + matmul):

1. Dump post-optimization HLO: `XLA_FLAGS="--xla_dump_to=/tmp/hlo --xla_dump_hlo_as_text"` or via [AOT compilation](aot-compilation.md)
2. Find the matmul (convolution) for the target projection: `grep 'convolution.*proj_name' *.after_optimizations.hlo`
3. Trace upward to its parent `fused_computation` entry
4. Check if that fusion's body includes the ops you planned to fuse (e.g., `rsqrt`, `multiply` for RMSNorm)
5. If yes → XLA already does it → retire the kernel-replacement hypothesis with `xla-already-fuses`

## When to use pass-by-pass dumps vs final-only

| Scenario | Recommended dump scope |
|---|---|
| Pre-filter a kernel-replacement hypothesis | **Final-only** (`after_optimizations`) — you only need to know the end state |
| Debug why a fusion broke between code versions | **Pass-by-pass** (`--xla_dump_hlo_pass_re='.*'`) — find which pass changed behavior |
| Diagnose a sharding problem | **SPMD partitioner** stage — see how GSPMD propagated shardings |
| Understand collective placement for LHS tuning | **Scheduler** stage — see the final op order |
| Offline analysis of customer-provided dumps | Whatever they provide — xprof-mcp tools work with any stage |

Pass-by-pass dumps can produce **hundreds of files** (one per pass per module). Use selective pass regexes to keep the output manageable.

## Conventions in this wiki

- HLO dumps are stored under `raw/profiles/<YYYY-MM-DD>-<exp-slug>/hlo/` (colocated with profile artifacts).
- The `.gitignore` has exceptions for `*-hlodump-*/` directories so evidence files are preserved.
- Experiment pages that use HLO inspection must cite the specific module file and stage in their `## Profile` or `## Sources` section.
