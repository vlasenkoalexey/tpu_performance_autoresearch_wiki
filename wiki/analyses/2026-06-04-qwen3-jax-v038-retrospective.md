# Retrospective: v038 (Splash + Fused Pre-Transpose RoPE)

## The Symptom
In experiment `v038`, we attempted to fuse RoPE and RMSNorm prior to transposing the tensor layout to eliminate the `data formatting` copy overhead observed in `v027`. The experiment failed immediately upon compiling the first training step. The TPU failed with:
`jax.errors.JaxRuntimeError: RESOURCE_EXHAUSTED: Ran out of memory in memory space vmem while allocating on stack for %splash_mha_dkv_no_residuals.1`
`Scoped allocation with size 34.90M and limit 32.00M exceeded scoped vmem limit by 2.90M.`

## Root Cause Analysis
The failure stems from strict compiler limits on the Pallas (custom kernel) VMEM stack. The custom Pallas Splash kernel is extremely tightly tuned to the specific memory layouts generated by JAX's default attention mechanisms. 

1. **Memory Layout Presumptions**: Splash anticipates specific contiguous strides when reading the `q, k, v` tensors.
2. **The Impact of Pre-Transpose Fusion**: By applying RoPE on the `[B, T, H, D]` tensor and *then* transposing to `[B, H, T, D]`, XLA successfully fuses the operations, but it alters the physical memory layout (strides) of the resulting tensors fed into the custom Pallas `splash` custom-call.
3. **VMEM Overflow**: When the `splash_mha_dkv_no_residuals` backward kernel attempts to compile, the layout mismatch forces XLA to emit additional on-device transposes/copies *inside the Pallas kernel block* to satisfy the operand layout constraints. These extra scratchpads push the scoped VMEM allocation for the kernel from its usual ~31M up to 34.90M. Since the physical hardware limit for scoped vmem is 32.00M on v6e, the compilation strictly fails.

## Conclusion and Next Directions
We cannot apply the `Pre-Transpose RoPE Fusion` optimization while simultaneously using Splash attention. Splash's tight VMEM limits and strict layout expectations preclude any upstream optimizations that alter the physical contiguity of its inputs.

### Path Forward: Batch Size Scaling
With `v038` falsified and `chunked_xla` (from `v037`) deprecated due to command queue stalling, we are exhausting algorithmic/fusion tricks on `TP=1`. 
However, in `v035` and `v036` (before we pivoted to chunked_xla), we observed that simply pushing `batch_size` upwards scales MFU consistently, though we hit HBM limits around `bs=24` (~34.6% MFU). 
The final play to achieve >35% MFU is to utilize standard Splash + dense XLA cross-entropy and **tune down the sequence length** from 8192/2048 to a smaller synthetic context to free up HBM, allowing us to dramatically scale up the local batch size (`bs=32` or `bs=64`). As the arithmetic intensity formula shows, scaling `B` drives MFU up by amortizing the static model weights cost across more tokens per step.
